## 프론트엔드에서 CS가 필요한 순간

프론트엔드 개발을 하다 보면 처음에는 화면이 보이는지가 가장 중요하다. 하지만 프로젝트가 커질수록 질문이 바뀐다.

- 왜 클릭 한 번에 화면이 버벅일까?
- 왜 리스트를 스크롤할 때 중간중간 끊길까?
- 왜 같은 값처럼 보이는데 React는 다시 렌더링할까?
- 왜 메모리 사용량이 계속 늘어날까?

이 질문은 단순히 React 문법만으로는 답하기 어렵다. JavaScript가 함수를 실행하는 방식, 객체를 메모리에 보관하는 방식, 브라우저가 렌더링 기회를 얻는 방식까지 같이 봐야 한다.

<Callout>
프론트엔드에서 CS는 시험용 지식이 아니라, 화면이 느려지는 이유를 설명하기 위한 언어다.
</Callout>

## 콜 스택은 지금 처리 중인 일의 순서표다

JavaScript는 기본적으로 한 번에 하나의 실행 흐름을 처리한다. 함수가 호출되면 콜 스택에 쌓이고, 함수가 끝나면 스택에서 빠진다.

```js
function handlePress() {
  validateForm();
  submitForm();
  updateUI();
}
```

이 코드는 짧아 보이지만, 각 함수 안에서 큰 배열을 순회하거나 무거운 계산을 하면 콜 스택이 오래 비워지지 않는다. 그동안 브라우저나 React Native 런타임은 다음 프레임을 그릴 기회를 늦게 얻는다.

사용자는 이 상황을 "앱이 멈췄다"라고 느낀다. 개발자는 이 상황을 "메인 스레드를 오래 점유했다"라고 설명할 수 있어야 한다.

## 힙 메모리는 객체가 살아 있는 공간이다

원시값은 비교적 단순하지만 객체, 배열, 함수는 힙 메모리에 저장되고 변수는 그 위치를 참조한다.

```js
const prev = { selected: true };
const next = { selected: true };

console.log(prev === next); // false
```

두 객체의 내용은 같지만 참조는 다르다. React에서 이 차이는 중요하다. props로 매번 새 객체를 넘기면, 자식 컴포넌트 입장에서는 값이 바뀐 것으로 보일 수 있다.

```tsx
<UserCard options={{ compact: true }} />
```

렌더링할 때마다 `options` 객체가 새로 만들어진다. 작은 화면에서는 문제가 안 보일 수 있지만, 큰 리스트나 비싼 자식 컴포넌트에서는 불필요한 렌더링 비용이 된다.

## 가비지 컬렉션은 공짜가 아니다

JavaScript는 사용하지 않는 메모리를 가비지 컬렉션으로 정리한다. 이 덕분에 직접 메모리를 해제하지 않아도 되지만, 객체를 너무 많이 만들고 버리면 정리 비용이 생긴다.

예를 들어 스크롤 중에 매 프레임 새 배열과 객체를 많이 만들면 다음 문제가 생길 수 있다.

- 짧은 시간에 많은 임시 객체가 생긴다.
- 가비지 컬렉션이 더 자주 필요해진다.
- 정리 타이밍이 프레임 렌더링과 겹치면 끊김이 보인다.

React Native에서 애니메이션이나 제스처가 민감한 이유도 여기에 있다. 손가락 움직임은 계속 들어오고, 화면은 일정한 간격으로 갱신되어야 한다. 이때 JS 스레드가 객체 생성과 정리로 바쁘면 인터랙션이 둔해진다.

## 렌더링 비용을 줄이는 순서

무작정 `memo`, `useMemo`, `useCallback`을 붙이는 방식은 오래 가지 않는다. 먼저 어떤 비용이 있는지 나눠야 한다.

1. 상태가 너무 위에 있어서 넓은 영역이 다시 렌더링되는가?
2. props로 매번 새 객체나 새 함수를 넘기는가?
3. 렌더링 중에 비싼 계산을 반복하는가?
4. 리스트 item의 key가 불안정한가?
5. 화면에 보이지 않는 영역까지 한 번에 렌더링하는가?

이 질문에 답한 뒤에 최적화를 적용해야 한다.

```tsx
const visibleUsers = useMemo(() => {
  return users.filter((user) => user.active);
}, [users]);
```

`useMemo`는 계산 결과를 기억하기 위한 도구다. 하지만 `users` 자체가 매번 새 배열이면 효과가 줄어든다. 결국 메모이제이션보다 먼저 데이터가 어디에서 새로 만들어지는지 확인해야 한다.

## 좋은 프론트엔드 개발자는 화면 뒤의 비용을 설명한다

성능 문제는 눈에 보이는 현상으로 시작하지만, 원인은 보이지 않는 곳에 있다. 콜 스택이 오래 비워지지 않는지, 힙에 객체를 너무 많이 만들고 있는지, 참조가 안정적인지, 렌더링 범위가 넓은지 확인해야 한다.

프론트엔드에서 CS를 안다는 것은 용어를 외우는 것이 아니다. 사용자가 느끼는 끊김을 실행 흐름과 메모리 비용으로 설명하고, 코드 구조를 바꿔 해결할 수 있다는 뜻이다.
