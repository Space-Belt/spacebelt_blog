## JavaScript 심화는 실행 순서를 의심하는 것에서 시작되었다

JavaScript를 처음 배울 때는 코드가 위에서 아래로 실행된다고 생각하기 쉬웠다. 하지만 실제 프론트엔드 개발에서는 그렇게 단순하지 않았다. 클릭 이벤트가 있고, API 요청이 있고, 타이머가 있고, Promise가 있고, React의 상태 업데이트가 있었다.

겉으로 보면 한 파일 안에 있는 코드지만, 실제로는 여러 줄의 일이 서로 다른 시간표를 가지고 움직였다. JavaScript를 깊게 이해한다는 것은 이 시간표를 읽는 일이었다.

<Callout>
버그는 문법을 몰라서만 생기지 않았다. 코드가 "언제" 실행되는지 잘못 예상할 때도 자주 생겼다.

## 호출 스택은 지금 당장 처리하는 일의 책상이었다

JavaScript 엔진은 함수를 실행할 때 호출 스택에 작업을 쌓는다. 호출 스택은 지금 당장 처리해야 하는 함수들의 목록이었다.

```js
function login() {
  validateUser();
  createSession();
}

function validateUser() {
  console.log("사용자 확인");
}

function createSession() {
  console.log("세션 생성");
}

login();
```

`login`이 먼저 호출 스택에 올라가고, 그 안에서 `validateUser`가 올라갔다가 내려왔다. 그 다음 `createSession`이 올라갔다가 내려왔다. 마지막으로 `login`도 끝나며 스택에서 내려왔다.

비유하면 호출 스택은 한 사람이 사용하는 책상과 같았다. 지금 펼쳐둔 일을 끝내야 다음 일을 펼칠 수 있었다. 책상이 하나이기 때문에 동시에 두 일을 실제로 처리하지는 않았다.

이 사실은 중요했다. JavaScript는 한 번에 하나의 동기 작업을 처리한다. 그래서 무거운 반복문이나 계산이 오래 걸리면 클릭, 스크롤, 화면 업데이트도 함께 늦어질 수 있었다.

```js
while (true) {
  // 이 코드는 브라우저 화면까지 멈추게 만들 수 있었다.
}
```

React 화면이 버벅이는 경우에도 결국 호출 스택을 오래 점유하는 코드가 있는지 의심해야 했다.

## Web API는 오래 걸리는 일을 맡아주는 대기실이었다

브라우저에는 JavaScript 엔진만 있는 것이 아니었다. 타이머, 네트워크 요청, DOM 이벤트 같은 기능은 브라우저의 Web API 영역에서 도와주었다.

```js
console.log("시작");

setTimeout(() => {
  console.log("타이머 완료");
}, 1000);

console.log("끝");
```

이 코드는 `시작`, `끝`, `타이머 완료` 순서로 실행되었다. `setTimeout`을 만났다고 JavaScript가 1초 동안 멈춰 있는 것이 아니었다. 타이머 작업은 브라우저에 맡겨지고, JavaScript는 다음 줄을 계속 실행했다.

비유하면 JavaScript 엔진은 카페의 계산대 직원이고, Web API는 주방이었다. 직원은 주문을 받고 주방에 음료 제조를 맡긴 뒤, 다음 손님의 주문을 계속 받았다. 음료가 완성되면 주방은 "이제 가져가도 된다"고 알려주었다.

## 태스크 큐와 마이크로태스크 큐는 대기 줄이 달랐다

비동기 작업이 끝나면 바로 호출 스택에 들어가는 것이 아니었다. 먼저 큐에 들어가서 기다렸다. 이때 모든 큐의 우선순위가 같지는 않았다.

대표적으로 `setTimeout`의 콜백은 태스크 큐에 들어가고, Promise의 `then` 콜백은 마이크로태스크 큐에 들어갔다. 마이크로태스크 큐가 태스크 큐보다 먼저 처리되었다.

```js
console.log("A");

setTimeout(() => {
  console.log("B");
}, 0);

Promise.resolve().then(() => {
  console.log("C");
});

console.log("D");
```

결과는 `A`, `D`, `C`, `B`였다.

처음에는 `setTimeout`이 0초니까 바로 실행될 것처럼 보였다. 하지만 0초는 "지금 바로 실행하라"는 뜻이 아니었다. "최소한 이 시간 이후에 태스크 큐에 넣어도 된다"는 뜻에 가까웠다.

마이크로태스크는 지금 동기 코드가 끝난 직후, 다음 태스크로 넘어가기 전에 먼저 비워졌다. 그래서 Promise의 `C`가 타이머의 `B`보다 먼저 출력되었다.

## 이벤트 루프는 호출 스택과 큐 사이의 신호등이었다

이벤트 루프는 호출 스택이 비었는지 확인하고, 큐에서 기다리는 작업을 호출 스택으로 옮기는 역할을 했다.

비유하면 이벤트 루프는 교차로의 신호등이었다. 호출 스택이라는 도로가 비어야 다음 차량이 들어올 수 있었다. 아직 도로 위에 차가 있으면, 큐에서 기다리는 작업은 아무리 급해도 들어오지 못했다.

```js
console.log("start");

setTimeout(() => {
  console.log("timeout");
}, 0);

for (let i = 0; i < 1_000_000_000; i += 1) {
  // 오래 걸리는 동기 작업
}

console.log("end");
```

타이머 시간이 0이어도 오래 걸리는 반복문이 끝나기 전까지 `timeout`은 실행될 수 없었다. 호출 스택이 비어야 이벤트 루프가 큐의 작업을 올릴 수 있기 때문이었다.

프론트엔드에서 긴 동기 작업을 쪼개야 하는 이유도 여기에 있었다. 사용자의 클릭과 스크롤도 결국 이벤트로 들어오는데, 호출 스택이 계속 막혀 있으면 사용자는 화면이 멈췄다고 느꼈다.

## 렌더링도 이벤트 루프 사이에서 기회를 얻었다

브라우저는 JavaScript만 실행하는 것이 아니었다. 스타일 계산, 레이아웃, 페인트도 해야 했다. 문제는 JavaScript가 호출 스택을 오래 점유하면 브라우저가 화면을 그릴 기회를 얻지 못한다는 점이었다.

```js
button.addEventListener("click", () => {
  button.textContent = "처리 중";

  heavyWork();

  button.textContent = "완료";
});
```

이 코드는 `처리 중`이라는 문구가 화면에 보이기 전에 `heavyWork`가 실행되어 버릴 수 있었다. JavaScript가 너무 오래 실행되면 브라우저는 중간 화면을 그릴 시간이 없었다.

그래서 사용자에게 즉시 피드백을 보여주고 싶다면 무거운 작업을 나누거나, 브라우저가 렌더링할 시간을 줄 필요가 있었다.

```js
button.textContent = "처리 중";

setTimeout(() => {
  heavyWork();
  button.textContent = "완료";
}, 0);
```

이 방식은 모든 상황의 정답은 아니지만, 렌더링이 JavaScript 실행 사이에서 기회를 얻는다는 사실을 이해하는 데 도움이 되었다.

React에서도 비슷한 감각이 필요했다. 상태를 바꾸었다고 즉시 DOM이 바뀌었다고 생각하면 안 되었다. React는 업데이트를 모으고, 계산하고, 브라우저 렌더링 타이밍과 함께 화면에 반영했다.

## 클로저는 편리하지만 메모리를 붙잡을 수 있었다

클로저는 함수가 생성될 당시의 외부 변수를 기억하는 기능이었다. 이 기능 덕분에 이벤트 핸들러, 커스텀 훅, 상태 관리 로직을 자연스럽게 만들 수 있었다.

하지만 클로저는 필요한 값만 기억하는 것이 아니라, 참조하고 있는 환경을 계속 붙잡을 수 있었다.

```js
function createHandler() {
  const largeData = new Array(100000).fill("data");

  return function handleClick() {
    console.log(largeData.length);
  };
}

const handler = createHandler();
```

`createHandler` 함수가 끝나도 `handler`가 살아 있는 동안 `largeData`는 사라지지 않았다. `handler`가 `largeData`를 계속 참조하고 있기 때문이었다.

비유하면 클로저는 필요한 자료를 가방에 넣고 다니는 기능이었다. 문제는 가방 안에 작은 메모지만 들어 있는 것이 아니라, 큰 책상까지 통째로 넣어버릴 수도 있다는 점이었다.

프론트엔드에서는 이벤트 리스너, 타이머, 구독 함수에서 이런 문제가 생길 수 있었다.

```js
function mount() {
  const data = new Array(100000).fill("item");

  window.addEventListener("resize", () => {
    console.log(data.length);
  });
}
```

이벤트 리스너를 제거하지 않으면, 화면에서 필요 없어졌다고 생각한 데이터도 계속 메모리에 남을 수 있었다. React에서 `useEffect`의 cleanup이 중요한 이유였다.

```tsx
useEffect(() => {
  function handleResize() {
    console.log("resize");
  }

  window.addEventListener("resize", handleResize);

  return () => {
    window.removeEventListener("resize", handleResize);
  };
}, []);
```

cleanup은 단순히 예의상 넣는 코드가 아니었다. 컴포넌트가 사라졌을 때 외부 세계에 걸어둔 연결을 끊는 작업이었다.

## 가비지 컬렉션은 닿을 수 없는 값을 치우는 과정이었다

JavaScript는 메모리를 직접 해제하지 않는다. 대신 가비지 컬렉터가 더 이상 접근할 수 없는 값을 찾아 정리한다.

```js
let user = {
  name: "Haja"
};

user = null;
```

처음에는 `user` 변수가 객체를 가리키고 있었다. 하지만 `user = null` 이후에는 그 객체에 접근할 방법이 사라졌다. 이런 값은 가비지 컬렉션 대상이 될 수 있었다.

중요한 기준은 "필요한가"가 아니라 "닿을 수 있는가"였다. 개발자가 보기에는 더 이상 필요 없어 보여도, 어딘가에서 참조하고 있으면 JavaScript는 그 값을 정리할 수 없었다.

```js
const cache = [];

function saveUser(user) {
  cache.push(user);
}
```

이런 캐시 배열에 계속 객체를 넣고 비우지 않으면, 객체는 계속 닿을 수 있는 상태로 남았다. 그래서 무한 스크롤, 이미지 목록, 검색 결과 캐시를 만들 때는 언제 데이터를 버릴지도 함께 설계해야 했다.

## 참조 동일성은 React 최적화의 출발점이었다

JavaScript의 객체와 배열은 참조로 비교된다.

```js
{} === {}; // false
[] === []; // false
```

겉보기에는 같은 빈 객체와 빈 배열이지만, 실제로는 서로 다른 주소를 가진 값이었다.

React에서 `memo`, `useMemo`, `useCallback`이 중요한 이유도 참조 동일성과 연결되어 있었다.

```tsx
function Parent() {
  const options = {
    sort: "latest"
  };

  return <ProductList options={options} />;
}
```

`Parent`가 다시 실행될 때마다 `options`는 새 객체가 되었다. 내용은 같아도 참조가 다르기 때문에, 자식 컴포넌트 입장에서는 props가 바뀐 것처럼 보일 수 있었다.

```tsx
function Parent() {
  const options = useMemo(() => {
    return { sort: "latest" };
  }, []);

  return <ProductList options={options} />;
}
```

이렇게 참조를 안정화하면 불필요한 렌더링을 줄일 수 있었다. 다만 `useMemo`를 무조건 붙이는 것이 좋은 코드는 아니었다. 참조 안정성이 실제로 필요한 곳인지 먼저 확인해야 했다.

## Promise 체인은 에러 흐름까지 설계해야 했다

Promise는 성공만 다루는 도구가 아니었다. 실패가 어디에서 잡히는지도 중요했다.

```js
fetch("/api/user")
  .then((response) => response.json())
  .then((user) => {
    console.log(user.name);
  })
  .catch((error) => {
    console.error(error);
  });
```

중간 `then`에서 에러가 발생해도 마지막 `catch`에서 잡을 수 있었다. 하지만 `catch`가 있다고 모든 문제가 해결되는 것은 아니었다. 실패했을 때 사용자에게 어떤 화면을 보여줄지, 다시 시도할 수 있게 할지, 이전 데이터를 유지할지까지 정해야 했다.

```tsx
if (isLoading) return <Loading />;
if (isError) return <RetryView />;
return <ProductList data={data} />;
```

프론트엔드에서 비동기 처리는 데이터 요청 코드만의 문제가 아니었다. 화면 상태 설계의 문제였다.

## JavaScript를 깊게 이해하면 디버깅 방식이 달라졌다

JavaScript 심화 개념은 면접용 지식으로만 필요한 것이 아니었다. 실제 디버깅 방식 자체를 바꿔주었다.

- 값이 이상하면 스코프와 클로저를 확인해야 했다.
- 실행 순서가 이상하면 이벤트 루프와 큐를 확인해야 했다.
- 화면이 늦게 반응하면 호출 스택을 오래 막는 작업을 찾아야 했다.
- 메모리가 계속 늘어나면 이벤트 리스너, 타이머, 캐시, 구독 해제를 확인해야 했다.
- React 렌더링이 많으면 참조 동일성과 상태 위치를 확인해야 했다.

결국 JavaScript를 깊게 공부한다는 것은 더 어려운 문법을 외우는 일이 아니었다. 코드가 실행되는 시간, 값이 살아 있는 범위, 브라우저가 화면을 그리는 기회를 이해하는 일이었다.

React와 React Native를 다루는 개발자에게 이 이해는 특히 중요했다. 사용자는 코드 구조를 보지 않지만, 실행 순서가 꼬이면 느린 화면을 보고, 메모리 관리가 부족하면 버벅임을 느끼고, 비동기 상태가 부족하면 불안정한 앱을 경험한다.

그래서 JavaScript의 깊은 이해는 화면 뒤에 숨은 기본기였다. 좋은 인터랙션과 안정적인 앱은 결국 이 기본기 위에서 만들어졌다.
