## 프레임 드랍은 단순히 애니메이션 문제가 아니다

React Native에서 드래그, 스와이프, 바텀시트 애니메이션을 만들다 보면 "왜 같은 코드인데 어떤 순간에는 끊길까?"라는 질문을 하게 된다.

겉으로는 애니메이션 문제처럼 보이지만, 안쪽에는 스레드와 스케줄링 문제가 있다. 운영체제는 CPU 시간을 여러 작업에 나눠준다. 앱 안에서도 JavaScript 실행, UI 렌더링, 네이티브 이벤트 처리, 네트워크 콜백이 각자의 타이밍을 가진다.

<Callout>
프레임 드랍은 화면이 못 그려진 사건이 아니라, 필요한 시간 안에 필요한 작업이 끝나지 못한 사건이다.
</Callout>

## 60fps는 약 16.67ms의 예산을 가진다

화면이 60fps로 부드럽게 보이려면 한 프레임을 약 16.67ms 안에 준비해야 한다. 이 시간 안에 입력 처리, 상태 계산, 레이아웃, 렌더링, 합성 같은 일이 끝나야 한다.

문제는 JavaScript 코드도 같은 시간 경쟁에 들어온다는 점이다.

```ts
function onPressFilter() {
  const result = largeItems
    .filter((item) => item.enabled)
    .map((item) => heavyNormalize(item))
    .sort(compareByScore);

  setItems(result);
}
```

이 작업이 버튼 클릭 한 번에는 괜찮아 보여도, 제스처 중이나 스크롤 중에 실행되면 프레임 예산을 쉽게 넘길 수 있다. 사용자는 이 순간을 "터치가 늦게 따라온다"라고 느낀다.

## 스케줄러는 공평하지만, UX는 공평하지 않다

운영체제 스케줄러는 여러 스레드에 CPU 시간을 나눠준다. 하지만 사용자 경험 관점에서는 모든 작업이 같은 우선순위가 아니다.

- 손가락을 따라가는 애니메이션
- 화면 전환
- 긴 리스트 필터링
- 로그 전송
- 캐시 정리

이 작업들이 같은 순간에 몰리면, 사용자는 가장 눈에 보이는 인터랙션의 지연을 먼저 느낀다. 그래서 UI 개발에서는 어떤 작업을 지금 처리하고, 어떤 작업을 나중으로 미룰지 판단해야 한다.

## React Native에서는 스레드 경계를 알아야 한다

React Native 앱은 단순히 JavaScript만 실행하는 구조가 아니다. 크게 보면 JS 쪽에서 상태와 로직을 처리하고, 네이티브 쪽에서 실제 화면과 플랫폼 기능을 다룬다.

중요한 포인트는 경계다.

- JS 스레드가 바쁘면 JS 이벤트 처리와 상태 업데이트가 늦어진다.
- UI 스레드가 바쁘면 화면 합성과 터치 반응이 늦어진다.
- 두 영역 사이 통신이 많아지면 작은 지연이 누적된다.

New Architecture, JSI, TurboModule 같은 개념이 중요한 이유도 여기에 있다. 단순히 최신 기술이라서가 아니라, JS와 네이티브가 데이터를 주고받는 비용과 구조를 바꾸기 때문이다.

## 프레임 드랍을 줄이는 실제 기준

프레임 드랍을 줄이려면 먼저 작업을 나눠야 한다.

1. 제스처 중에 꼭 필요한 계산인가?
2. 렌더링 전에 동기적으로 끝나야 하는가?
3. 화면 밖 데이터까지 지금 처리하고 있는가?
4. 네이티브 드라이버나 UI 스레드 기반 애니메이션으로 옮길 수 있는가?
5. 로그 전송이나 분석 이벤트는 사용자 인터랙션 뒤로 미룰 수 있는가?

예를 들어 드래그 중에는 위치 계산과 시각 피드백만 남기고, API 요청이나 무거운 정렬은 gesture end 이후로 미루는 것이 좋다.

```ts
function onGestureEnd() {
  snapToNearestPoint();
  requestAnimationFrame(() => {
    trackSheetInteraction();
  });
}
```

이 코드는 분석 이벤트를 없애는 것이 아니라, 사용자가 체감하는 중요한 작업 뒤로 미루는 예시다.

## 운영체제 지식은 앱 성능의 언어가 된다

OS를 공부한다는 것은 커널 개발자가 되기 위한 것만은 아니다. 프론트엔드 개발자에게도 스레드, 스케줄링, 동시성, 우선순위 개념은 앱 성능을 설명하는 언어가 된다.

React Native에서 프레임 드랍을 줄인다는 것은 단순히 애니메이션 코드를 예쁘게 쓰는 일이 아니다. 제한된 시간 안에 어떤 작업을 먼저 끝내야 하는지 결정하는 일이다.
