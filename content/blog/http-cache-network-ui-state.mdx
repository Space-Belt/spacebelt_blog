## 네트워크는 화면 밖에서 일어나지만, 사용자는 화면으로 느낀다

프론트엔드에서 API 요청은 코드 한 줄처럼 보일 때가 많다.

```ts
const response = await fetch("/api/events");
```

하지만 이 한 줄 안에는 DNS 조회, TCP 연결, TLS 협상, HTTP 요청, 서버 처리, 응답 다운로드, JSON 파싱이 들어 있다. 사용자는 이 과정을 보지 못한다. 대신 화면이 늦게 바뀌는지, 로딩이 멈춘 것처럼 보이는지, 다시 시도할 수 있는지로 네트워크 상태를 판단한다.

<Callout>
프론트엔드의 역할은 네트워크를 숨기는 것이 아니라, 네트워크의 불확실성을 사용자가 이해 가능한 상태로 번역하는 것이다.
</Callout>

## loading 하나로는 부족하다

초기에는 `isLoading` 하나로 충분해 보인다. 하지만 실제 서비스에서는 요청 상태가 더 복잡하다.

- 처음 데이터를 가져오는 중
- 이전 데이터는 있지만 새 데이터를 갱신하는 중
- 서버가 실패했지만 재시도 가능한 상태
- 권한이 없어서 실패한 상태
- 네트워크가 끊겨서 실패한 상태
- 캐시된 데이터가 오래된 상태

이 상태를 모두 같은 로딩 스피너로 보여주면 사용자는 지금 무엇을 기다리는지 알 수 없다.

```tsx
if (isLoading) return <FullPageSkeleton />;
if (isError) return <RetryPanel />;

return (
  <>
    {isFetching ? <SmallSyncBadge /> : null}
    <EventList events={data} />
  </>
);
```

처음 진입할 때와 이미 데이터가 있는 상태에서 다시 가져올 때의 UI는 달라야 한다. 기존 데이터를 유지한 채 작은 갱신 표시만 보여주는 편이 더 안정적이다.

## HTTP 캐시는 속도보다 신뢰의 문제다

캐시는 단순히 빠르게 보여주기 위한 장치가 아니다. 사용자가 같은 화면을 다시 볼 때 "아까 보던 내용이 유지된다"는 신뢰를 만든다.

HTTP 캐시에는 여러 기준이 있다.

- `Cache-Control`: 응답을 얼마나 재사용할 수 있는지
- `ETag`: 서버의 데이터가 바뀌었는지 비교하는 식별자
- `Last-Modified`: 마지막 변경 시각
- `304 Not Modified`: 내용이 바뀌지 않았으니 기존 응답을 재사용하라는 신호

프론트엔드에서 React Query 같은 도구를 쓸 때도 비슷한 생각이 필요하다. 모든 데이터를 매번 새로 가져오면 최신성은 좋아 보이지만, 화면은 불안정해질 수 있다. 반대로 너무 오래 캐시하면 사용자는 낡은 정보를 보게 된다.

## stale은 나쁜 상태가 아니다

`stale`은 "못 쓰는 데이터"가 아니라 "다시 확인할 필요가 있는 데이터"에 가깝다.

예를 들어 이벤트 목록 화면에서 30초 전 데이터를 보여주는 것은 대부분 괜찮다. 하지만 결제 상태나 입장 인증 상태는 더 빠르게 갱신되어야 한다.

```ts
useQuery({
  queryKey: ["events"],
  queryFn: fetchEvents,
  staleTime: 30_000
});
```

중요한 것은 모든 API에 같은 정책을 적용하지 않는 것이다. 데이터의 성격에 따라 stale time, refetch timing, retry 횟수를 다르게 봐야 한다.

## 실패 상태는 원인별로 나눠야 한다

네트워크 실패도 하나가 아니다.

- 400: 클라이언트 요청이 잘못됨
- 401: 인증이 필요함
- 403: 권한이 없음
- 404: 리소스가 없음
- 500: 서버 내부 오류
- timeout: 응답이 너무 늦음
- offline: 네트워크 연결 없음

이 차이를 UI에서 구분하지 않으면 사용자는 해결 방법을 알 수 없다. 인증 만료는 로그인으로 보내야 하고, 서버 오류는 재시도 버튼을 제공해야 하며, 권한 없음은 접근 불가 이유를 설명해야 한다.

## 좋은 네트워크 UI는 기다리는 시간을 설계한다

네트워크를 완전히 통제할 수는 없다. 대신 기다리는 시간을 설계할 수는 있다.

1. 처음 로딩에는 skeleton을 보여준다.
2. 기존 데이터가 있으면 화면을 유지하고 작은 갱신 표시만 보여준다.
3. 재시도 가능한 실패에는 명확한 retry 버튼을 둔다.
4. 인증 실패는 로그인 흐름으로 연결한다.
5. 오래 걸리는 요청은 중간 상태를 설명한다.

프론트엔드 개발자가 네트워크를 이해해야 하는 이유는 여기에 있다. HTTP와 캐시 정책을 알면 단순히 API를 호출하는 코드를 넘어, 사용자가 신뢰할 수 있는 화면 상태를 설계할 수 있다.
