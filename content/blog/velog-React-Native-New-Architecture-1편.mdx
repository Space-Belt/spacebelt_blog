> ReactNative로 API 호출을 연습하던 중, 한 스크린에서 많은 양의 데이터를 여러 컴포넌>트에서 호출을 하여 받은 데이터에 의존하는 것들을 화면에 표시를 하면, 빈 공간을 보는 경우>가 많아 따로 loading 하고 있는 화면을 보여줘야할 경우가 있었다.
>
이 문제가 어떤 이유로 발생하는건지, 해결방법은 없는지 찾아보다. 브릿지의 특성인 비동기 통신에서 발생한 문제라는것을 알게 되었고. 이와 같은 문제를 해결하기 위해 기존의 Architecture가 아닌`NewArchitecture` 를 계속 발전시키고 있다는것을 알게되어, 이 포스팅을 하게 되었다.


## 기존 아키텍쳐의 문제점들을 한번 알아보자.

### 1) 브릿지의 문제점

> 브릿지가 무엇인지에 대해서는, 이전에 RN 동작원리 포스팅에서 설명했으니 한번 보는것도 좋을것 같..다 ㅎ
> 

![image](https://github.com/user-attachments/assets/8c39ea89-e4a7-4843-919e-b57f77680d96)

#### 리엑트 네이티브의 스레드가 서로 소통하는데에는 브릿지가 사용되며, 이때 직렬화/역직렬화 과정을 거친다.
    - 자바스크립트스레드 → 브릿지  JSON으로 변환
    - 브릿지 → 네이티브스레드  네이티브 데이터 구조 객체로 변경
    - 그 반대로 다시 응답을 줘야할때도 똑같이 이런 현상이 발생한다.
    
    ⇒ 즉, 어떠한 형태던 각 스레드간의 소통이 많으면 오버헤드가 발생해 성능의 저하를 일으킨다.
    
#### 브릿지의 소통은 비동기 방식으로 고정되어 있다.
    - 이렇게 되면, 데이터를 주고 받는 과정에서 순서가 꼬이는 현상도 발생할수 있다.
    - 대량의 데이터를 전송하거나 복잡한 작업을 수행할 때 성능 저하를 초래한다
    - 우선순위를 두고 싶은 경우에도, 무조건 비동기 방식이기에 관리가 어렵다.
    
#### 스크롤이나 드래그드롭같은 이벤트가 있을 경우에는, 너무 많은 이벤트가 발생한다. 복잡한 처리를 수행할 경우 16.67ms 내에 1프레임을 생성해야하는데 이 시간을 많이 넘기면 프레임 드롭과 같은 일이 발생한다.

#### 단일 브릿지로 모든 소통을 해야한다.
   - 메시지 큐에 쌓이는 메시지들이 많아지면, 자바스크립트 스레드와 네이티브 스레드 간의 상호작용이 지연될 가능성이 있다. 즉, 병목현상이 생긴다. 큐에 쌓인 메시지들이 차례로 처리되기 때문에, UI 스레드에서의 작업이 지연될 수 있다.  (메시지 큐는 브릿지가 비동기 통신을 관리하는 방법이다 + 이벤트루프)

   - 자바스크립트 코드가 실행되는 동안 네이티브 모듈과의 통신이 같은 스레드에서 이루어지기 때문에, 자바스크립트 스레드가 복잡한 계산이나 대량의 데이터 처리를 수행하는 동안, 애니메이션이 부드럽지 않고 끊기는 현상이 발생할 수 있다.
    
#### **시작시 기본 모듈 로드**: 서로 다른 개체이기 때문에 JavaScript가 필요할 때 모듈을 시작할 수 없어 애플리케이션 시작 시 모든 기본 모듈을 로드해야 하는 문제가 있다.

- 기존에 자바스크립트에서 네이티브 모듈을 호출할때에는 필수적으로 Promise나 콜백을 지정해줬어야 했다. 이때 콜백이 있으면 콜백큐에 들어가서 나머지가 다 끝나야 이 콜백을 실행시켰다. 그래서 응답을 기다리는데 더 오래걸리기도 했고, 이는 콜백 지옥을 유발하기도 했다.

### 2) 비동기 특성의 고질적인 문제

- **복잡한 코드 흐름 - **
    
    비동기 통신은 코드 흐름을 복잡하게 만들 수 있다. 특히 콜백 지옥(callback hell)이나, 프로미스 체이닝(promise chaining)으로 인해 코드가 읽기 어렵고 유지보수가 힘들어질 수 있다. 이는 코드의 가독성을 떨어뜨리고, 디버깅을 어렵게 만든다.
    

    
- **동기화 문제 / 비동기 데이터 불일치 - **
    
    비동기 통신은 여러 작업이 동시에 실행되기 때문에, 데이터의 일관성을 유지하기 어려울 수 있다. 예를 들어, 두 개의 비동기 작업이 같은 데이터에 접근하여 수정할 경우, 데이터의 일관성이 깨질 수 있다. 이러한 동기화 문제는 버그를 유발할 수 있다.
    
    비동기 통신으로 인해 JavaScript와 네이티브 간의 데이터 상태가 일치하지 않을 수 있다. 이는 각 스레드가 서로 다른 시간에 데이터를 업데이트하기 때문이다.

> 

### 1) **JavaScript Interface (JSI)**

> **뉴아키텍쳐의 핵심!!!**
JSI는 브릿지의 문제를 해결하기 위해 도입된 인터페이스다.
	
- JSI는 C++로 작성된 인터페이스다.

- JSI 객체는 C++ 힙 메모리에 직접 생성되며, 자바스크립트 엔진은 이 객체에 대한 포인터를 가지게 된다. 이로 인해 자바스크립트와 네이티브 코드 간의 경계가 사실상 없어져 브릿지를 통하지 않고 통신이 가능해진다.

- JSI는 자바스크립트 객체와 네이티브 객체를 직접 참조할 수 있게 한다. 이는 C++ 코드가 자바스크립트 런타임에 직접 접근하여 네이티브 객체를 자바스크립트 객체로 노출할 수 있게 해준다. ( 이는 네이티브 코드와 자바스크립트 코드가 동일한 메모리 공간을 공유할 수 있음을 의미한다. )
    - 자바스크립트 런타임이란 ?  자바스크립트 코드가 실행되는 환경을 의미하며, 이 환경 내에서 자바스크립트 스레드가 동작한다. ( 자바스크립트 엔진, 메모리 힙, 콜 스택, Web APIs 로 이루어졌다 )

**⇒ 즉, 자바스크립트와 네이티브에서 서로의 코드를 직접적으로 사용할 수 있게 된다는 의미다.**

> “그렇다면, JSI가 도입됨으로써 RN은 어떤 이점들을 취할 수 있었을까?”
> 

1. 자바스크립트와 네이티브간의 직접적인 소통이 가능하다. (브릿지 필요 없어지므로, 직렬화 / 역직렬화 과정이 사라져 오버헤드를 방지한다.)

2. 자바스크립트 코드에서 JSI 객체를 호출하여 네이티브 함수를 실행 할 수 있다. 따라서 JSI 객체 호출은 자바스크립트 엔진 내부에서 직접 C++ 함수를 호출하므로, 네이티브에 가까운 성능을 제공하며, 

3. 동기적 호출이 가능해진다. ( 동기적 호출이 가능하기에 원하는 데이터를 바로 가져오거나, 원하는 것을 기다리지 않고 바로 반영할 수 있다)

4. JSI는 특정 자바스크립트 엔진에 종속되지 않는다. 이는 JSI가 자바스크립트 런타임에 대한 추상화 계층을 제공하기 때문이다. 이로 인해 개발자는 JSC, Hermes, V8 등 다양한 자바스크립트 엔진 중에서 알맞는 엔진을 선택하여 사용할 수 있다.

5. JSI는 타입 안정성을 제공한다. 이는 JSI가 C++로 작성되었기 때문에 가능한 일이다. C++의 강력한 타입 시스템을 활용하여, 자바스크립트와 네이티브 코드 간의 타입 불일치 문제를 방지할 수 있다. 또한, Turbo Modules는 JSI를 사용하여 타입 안전성을 더욱 강화해준다. (C++ 타입템플릿 사용)
