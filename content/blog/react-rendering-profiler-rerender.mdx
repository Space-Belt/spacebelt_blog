## 렌더링 최적화는 memo를 붙이는 일부터 시작하면 안 된다

React에서 화면이 느려졌을 때 가장 먼저 떠오르는 것은 `memo`, `useMemo`, `useCallback`이다. 그런데 최적화를 이 도구들부터 시작하면 꽤 자주 빗나간다.

컴포넌트가 다시 렌더링된다는 사실 자체가 항상 문제는 아니다. 문제는 다시 렌더링된 결과로 실제 사용자가 느낄 만큼 비싼 작업이 발생하는지다. 그래서 먼저 봐야 하는 것은 "몇 번 렌더링됐나"보다 "어디에서 시간이 쓰였나"다.

React DevTools Profiler를 켜면 렌더링 비용을 조금 더 구체적으로 볼 수 있다.

- 어떤 컴포넌트가 렌더링됐는지
- 각 컴포넌트가 렌더링에 얼마나 걸렸는지
- props 변경 때문에 렌더링됐는지
- 부모 렌더링에 딸려 렌더링됐는지
- 실제 DOM 반영이 어느 commit에서 일어났는지

이 정보를 보고 나서야 `memo`가 필요한지, 상태 위치를 옮겨야 하는지, 리스트 가상화가 필요한지 판단할 수 있다.

## render phase와 commit phase를 나눠서 본다

React 렌더링은 크게 render phase와 commit phase로 나눠서 볼 수 있다.

render phase에서는 React가 컴포넌트 함수를 실행하고, 이전 결과와 새 결과를 비교할 준비를 한다. 이 단계에서는 아직 DOM에 반영하지 않는다.

commit phase에서는 계산된 변경 사항을 실제 DOM에 반영하고, layout effect 같은 commit 이후 작업을 처리한다.

이 구분이 중요한 이유는 병목의 성격이 다르기 때문이다.

render phase가 느리다면 컴포넌트 함수 안에서 비싼 계산을 하거나, 너무 많은 자식이 함께 렌더링되거나, props identity가 매번 바뀌고 있을 가능성이 있다.

commit phase가 느리다면 실제 DOM 변경량이 크거나, layout effect에서 동기 layout을 유발하거나, 스타일 변경이 브라우저 렌더링 비용으로 이어지고 있을 수 있다.

즉 "React가 느리다"가 아니라 "render가 느린지, commit이 느린지"부터 나눠야 한다.

## 부모 상태 변경이 자식 렌더링으로 번지는 경우

아래 코드는 흔한 패턴이다.

```tsx
function ProductPage({ products }: { products: Product[] }) {
  const [keyword, setKeyword] = useState("");

  const filteredProducts = products.filter((product) =>
    product.name.toLowerCase().includes(keyword.toLowerCase())
  );

  return (
    <>
      <SearchInput value={keyword} onChange={setKeyword} />
      <ProductSummary products={filteredProducts} />
      <ProductList products={filteredProducts} />
    </>
  );
}
```

검색어가 바뀌면 `ProductPage`가 렌더링되고, `ProductSummary`, `ProductList`도 같이 렌더링된다. 여기까지는 자연스럽다.

문제는 `ProductList` 안에 1,000개의 아이템이 있고, 각 아이템이 이미지, 가격 계산, 이벤트 핸들러, 뱃지 조건을 모두 처리한다면 검색어 입력 한 글자마다 큰 비용이 생길 수 있다는 점이다.

이때 무작정 모든 컴포넌트를 `memo`로 감싸기보다 먼저 질문을 나눠야 한다.

- 입력할 때마다 전체 목록을 다시 계산해야 하는가?
- 목록 아이템의 props는 안정적인가?
- 화면에 보이는 아이템만 렌더링해도 되는가?
- 검색어 변경을 즉시 반영해야 하는가, 지연해도 되는가?

병목이 목록 렌더링이라면 `memo`보다 가상 리스트가 더 효과적일 수 있다. 병목이 필터 계산이라면 `useMemo`나 서버 검색이 더 맞을 수 있다. 병목이 입력 반응성이라면 `useDeferredValue` 같은 선택지도 생긴다.

## props identity가 매번 바뀌면 memo가 무력해진다

`memo`는 props가 이전과 같으면 렌더링을 건너뛸 수 있게 해준다. 하지만 props로 새 객체나 새 함수를 매번 넘기면 효과가 줄어든다.

```tsx
const ProductRow = memo(function ProductRow({
  product,
  style,
  onSelect
}: {
  product: Product;
  style: CSSProperties;
  onSelect: (id: string) => void;
}) {
  return (
    <button style={style} onClick={() => onSelect(product.id)}>
      {product.name}
    </button>
  );
});

function ProductList({ products }: { products: Product[] }) {
  return products.map((product) => (
    <ProductRow
      key={product.id}
      product={product}
      style={{ minHeight: 48 }}
      onSelect={(id) => console.log(id)}
    />
  ));
}
```

`ProductRow`를 `memo`로 감쌌지만 `style`과 `onSelect`가 매 렌더링마다 새로 만들어진다. 얕은 비교 기준으로는 매번 다른 props다.

이때는 props를 안정화할 수 있다.

```tsx
const rowStyle = { minHeight: 48 };

function ProductList({ products }: { products: Product[] }) {
  const handleSelect = useCallback((id: string) => {
    console.log(id);
  }, []);

  return products.map((product) => (
    <ProductRow
      key={product.id}
      product={product}
      style={rowStyle}
      onSelect={handleSelect}
    />
  ));
}
```

다만 이것도 무조건 해야 하는 것은 아니다. 자식 렌더링 비용이 작으면 identity를 안정화하는 코드가 오히려 복잡도만 늘릴 수 있다. Profiler에서 실제 비용을 확인한 뒤 적용하는 편이 낫다.

## key 문제는 성능보다 상태 버그로 먼저 나타난다

리스트에서 `key`를 index로 넣는 코드는 성능 글에서 자주 나온다. 그런데 실무에서는 성능보다 상태 버그로 먼저 터질 때가 많다.

```tsx
items.map((item, index) => <TodoItem key={index} item={item} />);
```

목록 중간에 아이템이 추가되거나 삭제되면 index가 밀린다. React는 key를 기준으로 이전 컴포넌트와 다음 컴포넌트를 대응시키는데, index가 바뀌면 엉뚱한 컴포넌트 인스턴스가 재사용될 수 있다.

그 결과 input 값, focus, 내부 state가 다른 아이템에 붙는 문제가 생길 수 있다.

```tsx
items.map((item) => <TodoItem key={item.id} item={item} />);
```

안정적인 id를 key로 쓰는 이유는 단순히 경고를 없애기 위해서가 아니다. React가 변경 전후의 항목을 정확히 추적하게 하기 위해서다.

## 상태를 아래로 내리는 것도 최적화다

최적화라고 하면 캐시나 memoization을 떠올리기 쉽지만, 상태 위치를 바꾸는 것이 더 효과적일 때가 많다.

```tsx
function Dashboard() {
  const [isOpen, setIsOpen] = useState(false);

  return (
    <>
      <HeavyChart />
      <ActivityTable />
      <button onClick={() => setIsOpen(true)}>필터 열기</button>
      {isOpen ? <FilterModal onClose={() => setIsOpen(false)} /> : null}
    </>
  );
}
```

필터 모달 열림 여부가 바뀔 때마다 `Dashboard` 전체가 다시 렌더링된다. `HeavyChart`와 `ActivityTable`이 비용이 크다면 모달 상태를 더 좁은 컴포넌트로 내려도 된다.

```tsx
function Dashboard() {
  return (
    <>
      <HeavyChart />
      <ActivityTable />
      <FilterButtonWithModal />
    </>
  );
}
```

이제 모달 상태 변경은 `FilterButtonWithModal` 주변에 갇힌다. 이 방법은 memo를 붙이지 않고도 렌더링 전파 범위를 줄인다.

## 최적화 순서는 이렇게 잡는 편이 좋다

React 렌더링 병목을 볼 때는 대략 이런 순서가 안전하다.

1. Profiler로 느린 interaction을 기록한다.
2. render phase와 commit phase 중 어디가 비싼지 본다.
3. 비싼 컴포넌트가 왜 렌더링됐는지 확인한다.
4. 상태 위치를 좁힐 수 있는지 본다.
5. props identity를 안정화할 필요가 있는지 본다.
6. 계산 캐시, `memo`, 가상 리스트, deferred update를 적용한다.
7. 다시 측정한다.

중요한 것은 최적화 도구를 외우는 것이 아니라, 비용이 발생하는 지점을 읽는 것이다.

`memo`를 잘 쓰는 사람보다, `memo`가 필요 없는 구조를 먼저 찾는 사람이 더 안정적이다. React 성능 최적화는 코드를 더 많이 감싸는 일이 아니라, 렌더링이 퍼지는 경로를 좁히고 실제 병목만 줄이는 일에 가깝다.
