## 1. 앱의 시작점 (진입점을 찾다) `index.js`

처음 Metro 서버가 시작되고, JavaScript 번들링을 위해 제일 먼저 하는일은 무엇일까?

메트로가 마냥 개발자가 작성한 코드에 모두 접근해야한다고 생각할뿐, 대체 어디서, 어떻게 접근하는건지 알지 못했다. 

하지만 모든일이 그렇듯 시작점이 있다!

그 시작점은 바로  `index.js` 이다. ( 무조건은 아니고 일반적으로 여기서 시작한다 )

### 근데 index.js?  이게 뭐였더라… 라는 생각이 들었기에 먼저 index.js 파일을 알아보자.

```jsx
import { AppRegistry } from 'react-native';
import App from './App';
import { name as appName } from './app.json';

AppRegistry.registerComponent(appName, () => App);
```

보통의 `index.js` 파일의 구성은 이렇다.

그럼 왜 이 파일이 진입점이 되는것일까?

**→ 앱을 구성할 루트컴포넌트를 정의하는곳이 바로 여기기 때문이다.**

그럼 여기서 어떻게 정의 한건데??

`AppRegistry.registerComponent(appName, () => App);` 

바로 이 함수로 보이는 코드에서, 이 앱을 구성할 루트 컴포넌트를 정의/등록한다.

루트 컴포넌트 → 즉, 최고로 위에있는 컴포넌트라는 뜻이다. 따라서 `metro bundler` 는 이곳으로 진입하여, 우리가 작성한 코드를 모두 탐색/분석 하여, 각 모듈의 의존성을 파악하고 식별한 결과를 하나의 `Javascript 번들` 로 만든다. 

+ 즉, 최상단부터 시작하여 차례대로 우리가 쓴 코드를 다 탐색한다고 생각하면 좋을것 같다.

+ React 에선 이와 비슷한 역할을 하는 곳으로 `ReactDOM.render(&lt;App /&gt;, document.getElementById('root'));` 가 있다.

**⇒ index.js라는 최상단에서 모두를 탐색해 하나의 코드로 된 Javascript 번들이 생성된다.**

## 2. 네이티브 애플리케이션의 초기화 / 브릿지 초기화

핵심! 

애플리케이션의 기본 UI를 처리하는곳은 네이티브 환경이다. 하지만 1번에서 번들링 된 우리의 `javascript` 코드가 왜 존재하느냐! 바로 이 자바스크립트 환경과 네이티브 환경이 서로 통신하며 UI 변경을 렌더링 해야하는 것이다, 따라서 이 둘간의 통신을 위한 환경을 만드는 과정이라고 생각하면 좋을것 같다.

> `RCTRootView` / `ReactRootView` 가 뭐야??

JavaScript로 작성된 UI를 네이티브 환경에 렌더링 하는 역할을 하는 아이들이다.


#### 그럼 여기서 일어나는 일에 대해 알아보자!
**
1. React Native 뷰를 생성 ( 즉, 네이티브 UI를 렌더링 해주는 아이를 생성해준다 )
**
iOS는 `AppDelegate.m`에서 `RCTRootView`를 생성(초기화)하여 React Native의 UI를 네이티브 뷰 계층에 추가한다. ( 다음, metro서버인 경우는 `RCTRootView`의 초기화 매개변수로 번들 URL 주소를 설정한다. ) 

Android는 `MainActivity.java` 또는 `MainActivity.kt`에서 `ReactActivity`를 사용해 `ReactRootView`를 설정하고, `ReactInstanceManager`를 통해 React Native 애플리케이션을 초기화한다. ( iOS와 같이 `ReactRootView`를 초기화할 때, `JSBundleLoader`를 설정하여 번들 URL을 지정한다. )

한번쯤, `AppDelegate.m` 와 `MainActivity.kt` 파일을 확인해보는 것을 추천한다!

1. **브릿지 설정**

> **브릿지란?**
>
**자바스크립트와 네이티브 모듈 간의 통신을 가능하도록 해주는 다리의 역할이다.**
> 
> - 자바스크립트와 네이티브 간의 상호작용: 브릿지는 자바스크립트 코드와 네이티브 코드 간의 비동기 메시지를 전달하며, 두 환경이 서로 데이터를 교환하고 기능을 호출할 수 있게 한다.
> - 네이티브 모듈 호출: 자바스크립트 코드에서 네이티브 모듈의 메소드를 호출하거나, 네이티브 측에서 자바스크립트 함수에 이벤트를 전달할 수 있다.
>     - 카메라, 위치 서비스 등 ReactNative에서 접근할 수 없는 모듈들을 자바스크립트에서도 호출할 수 있다는 것이다.

+ 브릿지의 문제 - 비동기 통신  (이에 대해선 다음 포스팅에서 알아보자)

UI를 위한 `RCTRootView` / `ReactRootView` 가 생성된 후 

- iOS - `RCTBridge` 가 초기화 되며,  `AppDelegate.m`에서 `RCTBridge`를 설정하고, 이 브릿지를 `RCTRootView`에 연결한다.
- Android - `ReactInstanceManager`를 초기화하여 자바스크립트와 네이티브 간의 브릿지를 설정하고, `ReactRootView`와 `ReactInstanceManager`를 연결한다.

** ⇒ 우리가 React Native로 작성한 코드는 결국 실제 렌더링 될때에 JavaScript 코드가 아닌 네이티브 코드로 렌더링 된다. 
따라서 이 과정은 JavaScript 로 작성한 코드의 UI를 네이티브 코드로 렌더링 해줄 수 있는 역할을 하는 아이(`RCTRootView` / `ReactRootView`)를 생성하고, 이 자바스크립트 환경과 네이티브 환경간의 소통을 위한 브릿지를 생성하는것이 핵심인 것이다.**

## 3. JavaScript 코드의 로드

2번 과정이 끝나 JavsScript 코드를 맞이할 준비가 끝났기 때문에, 이제 불러오고 시작을 해야할 단계다. 

1. **JavaScript 번들 요청 후 로드:** 초기화 작업이 끝난 후 `RCTRootView`(iOS) 또는 `ReactRootView` 가 `JavaScript` 번들을 로드하기 위해 `Metro 서버(개발중)` 또는 `로컬 파일 시스템(배포시)` 에서 번들을 요청한다. ⇒ 번들을 로드한다.

근데 번들이 어디있는지 어떻게 알고.. 요청하고 로드하는거야…? 아래에서 알아보자
#### 메트로 서버(개발중)라면 네이티브 애플리케이션의 초기화시에 주소가 지정되어있다.

iOS - `jsCodeLocation = [NSURL URLWithString:@"http://localhost:8081/index.bundle?platform=ios"];`
        
Anroid - `JSBundleLoader.createUrlLoader("http://localhost:8081/index.bundle?platform=android");`
        
#### 배포된 앱 사용시
        
iOS - 앱의 리소스 번들 내에서 파일을 로드 `JSBundleLoader.createFileLoader(new File(getFilesDir(), "index.android.bundle"))`
        
Android - APK의 `assets` 폴더에서 파일을 로드
        
    
**⇒ 아까 2번째 과정에서 네이티브 애플리케이션의 초기화 / 브릿지 초기화 부분에서 React Native 뷰 생성 파트에서 잠깐 다뤘지만, 바로 이곳에서 주소가 지정되었기에 가능한것이다!**

이미지를 한번 간단하게 설명하고 각 스레드를 알아보도록 하겠다.

아까 매그도나르도로 설명했듯, 각각의 스레드는 각각의 서로 다른 역할을 배정받아 일을 한다. 나의 스레드 예시에선 사람으로 설명했기에 서로 말과 행동으로 소통하겠지만, 리엑트 네이티브의 스레드끼리의 소통은 json형식 으로 이루어진다.

그럼 아래에서 각 역할을 알아보자.

### 1) 메인스레드 (Main Thread)

주역할 - 최종적으로 유저에게 보여지는 화면을 구성해주는 역할이다. ( 이는 RCTRootView, ReactRootView 를 통해 유저에게 보여진다. )

건물을 만들때, 누군가 도면을 그리고 어떤 재료가 들어가는지 계산하고 설정하는 작업을 해서, 이 아이에게 전달이 되면 그것을 만들어 사람들에게 보여지게 하는 역할을 하는것이라고 생각하면 좋을것 같다.

### 2) 백그라운드스레드 (Shadow Thread)

주역할 - UI 레이아웃과 관련된 계산을 한다! 레이아웃을 정의하고 배치하는 일을 한다! 음… 화면에 그려질 모든 컴포넌트의 너비,높이,여백,패딩 등의 레이아웃 값들을 정의하고 배치하는것이다!

이 아이는 건물의 크기가 얼만큼이 될것인지, 창문의 크기는 어떨지, 입구는 몇개인지, 그리고 창문과 입구는 어디어디에 있고 출구는 어디에 있는지 계산하는 역할을 한다.

쉽게 생각하면 사람들에게 보여지는 것들의 즉, 요소의 위치, 크기 등이 어떻게 될지 계산하여, 메인스레드에게 전달해주는 역할을 한다고  생각하면 좋을것 같다.

> 이때, 이 계산 작업은 위 이미지에서 볼 수 있듯, 백그라운드 스레드에 있는 Yoga라는 리엑트 네이티브의 레이아웃 엔진이 진행한다. 
>
Yoga의 핵심은 CSS Flexbox 모델을 구현한 엔진이라는 것이다. 
>
어렵게 생각할것 없이 리엑트 네이티브 뷰를 그리다보면 알겠지만, 한 스크린을 구성할때 자동적으로 `flexDirection: column` 이 적용된것처럼 그려진다는것을 느꼈을것이다. 그 이유가 바로 Yoga가 Flexbox 레이아웃 모델을 구현했기에 그런것이다! ( 부모 컨테이너의 속성과 자식 요소의 속성에 따라 요소의 위치와 크기가 자동으로 조정된다 )
>

    Yoga의 백그라운드 스레드에서의 작동 방식
    
    1. **Shadow Nodes 생성 및 관리**:
        - React Native 애플리케이션에서 각 UI 컴포넌트는 그에 대응하는 **Shadow Node**를 갖습니다. Shadow Node는 컴포넌트의 레이아웃 속성(예: 너비, 높이, 여백, 패딩 등)을 저장하는 역할을 합니다.
        - 백그라운드 스레드에서, 각 Shadow Node는 Yoga Node를 포함하고 있으며, 이 Yoga Node는 실제 레이아웃 계산을 수행합니다.
    2. **Props 전달 및 레이아웃 계산**:
        - React Native에서 UI 컴포넌트의 속성(props)이 변경되면, 이러한 변경 사항은 백그라운드 스레드로 전달됩니다. 이 때, Yoga는 변경된 속성을 기반으로 레이아웃을 재계산합니다.
        - 예를 들어, Flexbox 레이아웃 속성(flexDirection, justifyContent 등)이 변경되면, Yoga 엔진은 이 정보를 사용하여 각 컴포넌트의 크기와 위치를 다시 계산합니다.
    3. **비동기 처리**:
        - Yoga는 비동기적으로 레이아웃을 계산합니다. 즉, 메인 스레드와 독립적으로 레이아웃 계산이 이루어지기 때문에, UI가 복잡한 레이아웃 계산으로 인해 블로킹되는 일이 줄어듭니다.
        - 이 비동기 처리 덕분에 애플리케이션의 사용자 인터페이스는 더 부드럽고 빠르게 반응할 수 있습니다.
    4. **레이아웃 결과 전달**:
        - 레이아웃 계산이 완료되면, Yoga는 그 결과를 Shadow Node에 저장합니다. 이 Shadow Node는 다시 메인 스레드로 해당 정보를 전달합니다.
        - 메인 스레드는 이 정보를 사용하여 실제 UI 요소를 렌더링합니다. 예를 들어, 컴포넌트의 정확한 위치와 크기를 결정하여 화면에 표시합니다.

### 3) 자바스크립트 스레드 (JavaScript Thread)

주역할 - 애플리케이션의 상태(state)변화, 유저가 발생시킨 이벤트처리, 네트워크 요청 등 전체적인 로직을 담당한다!

즉, 건물의 엘리베이터가 움직이는것, 건물의 외관 조명이 언제 켜지고, 어느색으로 변하고, 어떤 사람이 창문에 돌을 던지면 그 창문에 어떤일이 일어날지, 재난이 일어나면 입구는 어떻게 봉쇄되는지 등을 계산하여, 메인스레드에 보내주는 역할을 하는것이라고 보면 될것 같다.

 

### 그럼 스레드간의 상호 작용을 알아보자 ( 이는 브릿지를 통해 서로 소통하는 것이다 )

**메인 → 자바**

- 메인에서 발생한 이벤트나 데이터를 자바스레드에 전달한다.
- 메인은 이벤트 리스너(?) 라고 생각하면 좋을것 같다. 이벤트가 발생하면 브릿지에게 이벤트가 발생됐다고 알리고, 이 브릿지는 자바스크립트 스레드에 어떤 이벤트가 발생했다고 알려준다.
- 예 - 누군가 건물에 돌을 던졌고, 그 돌이 창문에 닿았다. 이후에 일어날 일들을 자바스레드라는 아이에게 알려 어떠한 로직으로 어떤 변화가 생길지는 모르겠지만, 이 일을 알린다.

**자바 → 백그라운드**

- 자바스레드에서 로직이나 상태값을 변경하면서 레이아웃이 변경되었는지 계산하도록 한다.
- 예 - 건물에 누군가 돌을 던졌고, 그 돌이 너무 컸던 탓에 1층에 있던 창문유리만 깨지는 것이 아니라 외벽이 무너져 내리고 안의 근처에 있던 기계 등이 망가졌다. 이 상황으로 인해 건물을 바라보던 사람들에게 어떤 변화가 있는지를 계산하라고 보내주는 것이다.

**백그라운드 → 메인**

- 레이아웃이 변경됐다면 메인스레드에 계산된 UI를 보내고 메인은 자기역할을 다함
- 예 - 건물이 맞은 돌로 인해, 1층에 있던 창문이 깨지고 외벽이 허물어져 외부에서 봤을때 2개의 방이 훤히 보이게 됐음을 계산하여, 이 정보를 사람들에게 보여줄수 있는 메인스레드에게 전달하는것이다.

**자바 → 메인**

- 자바스레드에서 이벤트 처리하고, 그에 따른 상태의 변화가 있으면 메인스레드로 보내 UI를 업데이트 한다.
    - 텍스트인풋이라 가정했을때 텍스트 인풋의 레이아웃은 변경되지 않고 그 안의 상태값만 변경 됐을때.
- 예 - 위 예시들과 다르게 돌을 던졌지만 하나의 창문에 기스가 난것이 끝이었다. 이 변화는 건물의 전체적인 규격, 세부적인 규격을 건들지 않았고, 스크레치가 조금 났다. 이때, 메인스레드에 창문의 상태가 변했다고 알려주는 역할을 한다.

예시를 보고 더 어지러운 일이 발생하지 않으면 좋겠다…

**⇒ 즉, 세가지 다른 동작을 하는 스레드가 있고, 이 스레드들끼리는 서로 브릿지로 상호작용을 한다. 이제 마무리를 향해 가보자.**

## 4. JavaScript 코드의 실행

#### 자바스크립트 번들이 실행되는 과정을 알아보자!!

1. JavaScript 스레드가 번들을 실행하면서, 번들에도 존재하는 엔트리포인트(index.js)호출하여 실행한다.

1. 1번을 통해 JavaScript 스레드가 애플리케이션의 초기 상태를 설정하고 컴포넌트 트리를 생성하며, 이 과정에서 리엑트 핵심 개념인 `Virtual Dom` 이 생성된다. 
    - 컴포넌트 트리간단 설명
        
        리엑트는 모든것이 부모 → 자식 (n개)→ 자식(n개) → 자식(n개) 이렇게 구성된다. 따라서 index.js의 Root인 App(부모) → 자식(n개) → 자식(n개) → 자식(n개)  이렇게  계층 구조를 가지는데 이거를 나무 형태로 생성한다고 생각하면 될것 같다.
        
    - Virtual Dom ??  ⇒ JavaScript 스레드에서 UI의 상태와 구조를 관리하는 역할 ( JavaScript 스레드에서 관리됨 )
    
2. 2번을 바탕으로 백그라운드 스레드에 전달할 레이아웃 계산 명령이 생성됐고, 이제 브릿지를 통해 이 명령을 백그라운드 스레드에 전달한다.
    
    이 명령들은 Virtual Dom을 바탕으로 생성되며 아래와 같은 내용을 포함한다.
    
    - **UI 요소의 생성**: 새로 추가할 UI 요소와 그 속성
    - **UI 요소의 업데이트**: 기존 요소의 업데이트된 속성 및 스타일
    - **UI 요소의 삭제**: 제거할 요소의 정보
    - **스타일 및 레이아웃 정보**: 요소의 위치, 크기, 정렬 및 스타일 정보
    - **애니메이션 및 트랜지션 정보**: 애니메이션 효과 및 상태 전환

1. 백그라운드 스레드에서 브릿지로 명령을 받아,  Yoga가 레이아웃 관련 스타일 속성(크기, 여백, 패딩, 정렬 등)을 계산한다. 레이아웃과 관련되지 않은 스타일들은 자바스크립트 스레드에서 정의되어 넘어오기에, Yoga가 건드리지 않는다.

1. 이제 UI에 그려져야할 정보들이 모두 정의 되었다면, 메인스레드에게 브릿지를 통해 전달한다.

1. 전달받은 Json 형태의 UI 최종본을 메인스레드에서 `RCTRootView` / `ReactRootView`  를 통해 유저가 보는 스크린에 그려낸다.

여기까지는 앱이 제일 처음 실행 시의 상황이다. 그 후로는 사용자의 Action이 들어갈 수 있는것이다. 버튼의 클릭이던 textinput의 값을 입력했건 어떠한 이벤트가 발생한다. 

그럼 이 상황에서의 진행은 어떻게 될까? 

1. 예를 들어, 버튼 클릭, 텍스트 입력, 스크롤 등과 같은 유저 액션이 발생한다. 이때 이 이벤트는 메인스레드(네이티브 스레드)에서 처리 된다. 
    - **시스템 이벤트**: 예를 들어, 네트워크 상태 변화, 화면 회전, 또는 메모리 경고 등의 시스템 이벤트도 포함된다.
    - 로직이 실행되는것이 아니다.
    - 처리한다는것은 유저가 어떤 동작을 했는지 담아서 데이터화/캡쳐 하여 자바스크립트 스레드에서 이해할수 있는 형태로 변환하는것을 의미하지 / 바로 화면에 적용시키는 것이 아니다.
    
2. 이제 유저의 액션이 담긴 것을 자바스크립트 스레드로 브릿지를 통해 전달한다.

1. 자바스크립트 스레드에서 전달받은 이벤트 데이터를 수신하고, 일어난 이벤트에 대한 이벤트 핸들러를 실행시킨다.  즉, 해당 이벤트에 대한 로직을 실행한다. 

1. 각각에 맞는 이벤트 핸들러가 실행됐고, 이에 따른 애플리케이션의 상태에 업데이트가 일어났다면, 이때 새로운 상태변화로 인한 Virtual Dom을 새로 생성하고 기존의 Virtual Dom과 비교하여 변경된 부분을 찾아낸다.
    - React는 가상 DOM 비교(Virtual DOM diffing)를 통해 변경된 부분을 감지
2. 변경된 부분을 찾았다면 자바스크립트 스레드에서 백그라운드 스레드에 렌더링 명령을 내린다.

1. 백그라운드 스레드는 변경된 부분을 반영하여 레이아웃 스타일 변경 계산을 하고, 이 데이터를 메인 스레드에게 보낸다.

1. 메인스레드는 변경된 사항을 반영하여 유저에게 화면을 그려준다.

## 요약

**React Native의 기존 아키텍처에서 UI가 렌더링되는 과정은 다음과 같다:**

1. JavaScript 번들 실행 (JavaScript Thread)
    - 앱이 시작되면 JavaScript 번들이 JavaScript 스레드에서 실행된다.
    - 이 스레드는 React 컴포넌트 로직을 실행하고 UI 구조를 정의한다.
2. React Element Tree 생성 (JavaScript Thread)
    - React는 컴포넌트 로직을 실행하여 React Element Tree를 생성한다.
    - React Element Tree는 순수한 JavaScript 객체로, UI의 구조와 속성을 표현한다.
    - 이 트리는 React 컴포넌트의 계층 구조를 나타내며, 각 노드는 컴포넌트의 타입, props, 자식 요소 등의 정보를 포함한다.
3. React Shadow Tree 생성 (Shadow Thread)
    - React Element Tree를 기반으로 React Shadow Tree가 생성된다.
    - React Shadow Tree는 레이아웃 계산을 위한 중간 표현으로, C++로 구현되어 있다.
    - 각 Shadow Node는 React 컴포넌트에 대응하며, 레이아웃 관련 정보를 포함한다.
    - Shadow Tree는 JavaScript와 네이티브 플랫폼 사이의 브릿지 역할을 하며, 효율적인 레이아웃 계산을 가능하게 한다.
4. 레이아웃 계산 (Shadow Thread)
    - Yoga 레이아웃 엔진을 사용하여 React Shadow Tree의 각 노드에 대한 레이아웃을 계산한다.
    - 이 과정에서 각 컴포넌트의 크기와 위치가 결정된다.
5. 네이티브 UI 업데이트 명령 생성 (Shadow Thread)
    - 레이아웃 계산 결과를 바탕으로 네이티브 UI를 업데이트하기 위한 명령을 생성한다.
6. 네이티브 UI 업데이트 (Main/UI Thread)
    - 생성된 업데이트 명령이 Main/UI 스레드로 전달된다.
    - 이 스레드에서 실제 네이티브 UI 컴포넌트가 생성되거나 업데이트된다.
7. 화면에 렌더링 (Main/UI Thread)
    - 최종적으로 업데이트된 네이티브 UI 컴포넌트가 화면에 렌더링된다.

**주요 개념 요약:**

- **JavaScript Thread**: React Native 앱의 JavaScript 코드가 실행되는 스레드다. React 컴포넌트 로직과 상태 관리가 이루어진다.
- **Shadow Thread**: 레이아웃 계산과 UI 업데이트 준비를 담당하는 백그라운드 스레드다. React Shadow Tree를 생성하고 관리한다.
- **Main/UI Thread**: 실제 네이티브 UI 컴포넌트를 생성하고 화면에 렌더링하는 스레드다. 사용자 상호작용도 이 스레드에서 처리된다.
- **React Element Tree**: React 컴포넌트의 렌더링 결과를 나타내는 가벼운 JavaScript 객체 트리다. 각 요소는 컴포넌트의 타입, props, 자식 요소 등을 포함한다. 이 트리는 UI의 구조를 설명하지만 실제 DOM이나 네이티브 뷰와는 직접 연결되지 않는다.
- **React Shadow Tree**: UI의 구조와 레이아웃 정보를 담고 있는 중간 표현이다. JavaScript와 네이티브 플랫폼 사이의 브릿지 역할을 한다. C++로 구현되어 있어 빠른 레이아웃 계산이 가능하다. Shadow Tree는 React Element Tree의 각 요소에 대응하는 노드를 가지며, 이 노드들은 레이아웃 계산에 필요한 추가 정보를 포함한다.
- **Yoga 레이아웃 엔진**: Facebook에서 개발한 크로스 플랫폼 레이아웃 엔진으로, Flexbox 스타일의 레이아웃 계산을 수행한다.

이 과정을 통해 React Native는 JavaScript로 작성된 UI 로직을 네이티브 플랫폼의 실제 UI 컴포넌트로 변환하고 렌더링한다.
