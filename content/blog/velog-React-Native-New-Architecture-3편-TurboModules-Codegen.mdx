## TurboModules가 뭐야?

>TurboModules는 JavaScript와 네이티브 코드 간의 효율적이고 타입 안전한 직접 통신을 가능하게 하는 React Native의 새로운 모듈 시스템이다.

#### TurboModules의 핵심 개념도 JSI 다. 


내가 중요하다고 생각하는 순서대로 TurboModules를 정리해보겠다.



#### 1)  기존 아키텍처 문제점

**문제**  

기존 아키텍처에서는 자바스크립트와 네이티브는 서로 완전히 다른 존재기에, 사용하려면 앱 시작시에 모든 네이티브 모듈을 한번에 로드했어야 했다. 또 브릿지가 있었기에 오버헤드가 일어나는건 물론. 메모리도 많이 차지하고 성능적으로 좋지 않다.

**해결**

새로운 아키텍처 TurboModules는 지연 로딩을 채택했으며, 방식은 아래와 같다.

1. **앱 시작 시**:
    - Codegen이 모듈의 인터페이스를 자동으로 생성하고 정의한다.
    - 모든 모듈은 초기화되지 않은 상태로 남아 있다.
2. **JavaScript에서 모듈 호출 시**:
    - JSI가 Codegen이 정의한 인터페이스를 통해 모듈 호출을 처리한다.
    - JSI가 해당 모듈이 로드되었는지 확인한다.
    - 모듈이 로드되지 않은 경우, JSI가 모듈을 동적으로 로드하고 초기화한다.
3. **모듈 초기화 후**:
    - 초기화된 모듈은 이후 호출에서 바로 사용 가능하다.

이를 가능케 하는 TurboModules에 핵심 기술은 JSL과 Codegen이다. 그럼 둘이 어떻게 작동했길래 지연로딩을 할수 있었는지 알아보자

**1) Codegen의 역할**:

- Codegen은 JavaScript 명세를 기반으로 네이티브 코드 인터페이스를 자동으로 생성하고 정의한다.
- 이 인터페이스 정의는 JavaScript와 네이티브 코드 사이의 "계약"과 같은 역할을 합니다. ⇒ 참조할수 있도록 해주는것.

**2) JSI의 동작**:

- JSI는 Codegen이 생성한 이 인터페이스 정의를 사용하여 JavaScript에서 네이티브 모듈을 호출할 때 어떤 모듈을 찾아야 하는지, 어떤 메서드를 호출해야 하는지 알 수 있다.
- 인터페이스 정의가 없다면, JSI는 어떤 모듈을 찾아야 하는지, 그 모듈이 어떤 메서드를 가지고 있는지 알 수 없다.

**3) 모듈 로드 확인 과정**:

- JavaScript에서 모듈을 호출할 때, JSI는 먼저 Codegen이 정의한 인터페이스를 참조한다.
- 이 인터페이스를 통해 JSI는 해당 모듈이 존재해야 하는지, 어떤 구조를 가져야 하는지 알 수 있다.
- 그 다음 JSI는 이 정보를 바탕으로 모듈이 실제로 로드되었는지 확인한다.

**4) 동적 로딩**:

- 모듈이 로드되지 않았다면, JSI는 인터페이스 정의를 참조하여 어떤 모듈을 어떻게 로드해야 하는지 결정합니다.
- 인터페이스 정의 없이는 이 과정이 불가능합니다.

**문제**

기존의 아키텍쳐에서는 브릿지의 비동기 통신을 이용해서 즉시 결과가 필요한 작업에서 불필요한 지연이 발생했다. 또, 복잡한 로직을 구현할 때 콜백 지옥이나 프로미스 체인이 생겨 코드 복잡도가 증가했고, 이는 여러가지 성능저하를 이루어냈다. 대표적으로는, 유저에게 표시되는 애니메이션에 프레임드롭이 생겨, 불편을 만들어내기도 했다.

**해결** → TurboModules의 동기적 메서드 호출 기능 

**설명**

1. 처음 로드되면 Codegen이 모듈 인터페이스를 정의했고, 자바스크립트가 호출을 하면 JSL이 이 인터페이스를 통해 네이티브에 모듈이 준비되었나 확인 후 없으면 로드하고 초기화를 한다. 
2. 실행 과정 
    
    자바스크립트가 동기적으로 모듈 호출 한다. → Codegen이 정의한 모듈 인터페이스를 통해 JSI가 모듈이 로드되었는지 확인 후 없으면 로드 하고 초기화를 한다 → 이제 자바스크립트가 이 모듈을 사용할 수 있는것이다. 
    
    **⇒ 동기적으로 호출을 했기에, 이 응답을 받기전에 자바스크립트는 호출 다음에 할일이 있어도 자바스크립트 런타임에 의해 잠깐 작동이 블록킹 된다. 하지만 즉시 결과를 받을 수 있다. **
    
    여기서 JSI로 인해 성능을 향상 시켰기에 한번 확인해보자
    
    - 동기적 호출은 불필요한 비동기 래퍼를 제거하여 성능을 향상시킨다.
        
        ⇒ 원래는 프로미스나, 콜백으로 감싸야 했는데, JSI로 인해 따로 처리가 없어지기에 메모리상의 여유가 생기고 이는 성능향상을 나타낸다.
        
    - 즉시 결과가 필요한 작업(예: 애니메이션, 레이아웃 계산)에서 특히 유용하다.
        
        ⇒ 동기적 작업으로 먼저 처리해야하는 애니메이션을 처리해 놓으면, 다른 작업을 기다리지않고 이를 먼저 하게하면 빠르게 프레임을 구성하여 프레임 드롭이 줄어들어 부드러운 사용자 경험을 제공할 수 있다.

**문제**

기존 아키텍처에서는 iOS와 Android 간의 네이티브 코드가 서로 완전히 다른 존재로 취급되었다. 그래서 iOS와 Android는 똑같은 일을하는 코드여도 별도로 구현이 필요했다. 당연히 유지보수도 두번을 해야했고. 동일한 기능을 두 번 구현해야하는데 언어가 다르기에 일관성을 유지하기도 힘든 문제가 있었다. 

**설명** 

- 이 또한 앞서 설명했던 로직과 거의 비슷하지만 여기는 한가지가 더 필요하다.
- 그것이 뭐냐면, IOS, ANDROID가 둘다 이해하는 코드를 작성하기 위해선 C++ 언어로 같은 일을 하는 코드를 작성해야하는 것이다. ( 파일명.cpp )
- 그럼 전에 해결했던 문자들과 같이 앱이 실행되면서 Codegen이 C++기반으로 모듈에 관한 Interface를 작성하고 JSI를 통해 똑같이 적용되는것이다.
- 여기서 주의할 점은, iOS: Objective-C++ / Android: **JNI (Java Native Interface)** 를 통해 개발자가 작성한 C++언어를 쓸 수 있도록 연결해준다.  ( 즉, 플랫폼 별 브릿지 역할을 하는 아이들이 연결해준다는 것을 알면 된다! )

⇒ 여기까지 TurboModules를 봤다면 무엇인가 느껴졌을것이다. TurboModules은 JSI와 Codegen을 활용하여 작업들을 이루어내는 NewArchitecture의 구조물의 이름중 하나인것 같다. (..제가 생각하기엔 그렇습니다.)

4) Codegen

> 사실 Codegen이 스스로 뭘 하는게 아니라 new Architecture들과 상호작용해야 효과를 낼 수 있는 아이다.

#### Codegen가 스스로 할수 있는 일은 이것이다.

**Codegen**은 **JavaScript/TypeScript** 명세를 기반으로 네이티브 코드 인터페이스를 자동으로 생성하여, 개발자가 직접 작성해야 하는 코드의 양을 줄이고 타입 안전성을 보장합니다.

** => 사실 이건 JSI가 없으면 그냥 혼자 덩그러니 인터페이스만 만들고 가만히 있는것이다. **

하지만 이대로 끝내기는 좀 아쉬우니 다루지 않았던 Fabric과 Codegen이 만나면 어떻게 되는지 잠깐 알아보자!

> Codegen과 Fabric이 만나면 ??

지금까지 설명한대로면 JavaScript와 네이티브 모듈은 서로 호출해서 서로의 코드를 JSI덕에 편하게 사용할 수 있었다. 

**[TurboModules]** 을 설명하면서 했던 **Codegen**이 인터페이스를 정의하는파트가 있다. 

네이티브 개발자가 iOS에서 커스텀 버튼을 만들었다고 생각해보자. 이거를 JavaScript에서 사용할 수 있는 방법이 있을까??  

놀랍게도 Codegen과 Fabric이 만나면 이것이 가능하다.
