## 상태 관리는 라이브러리 선택보다 경계가 먼저다

프론트엔드에서 상태 관리를 이야기하면 Redux, Zustand, Recoil, Jotai, React Query 같은 도구 이름이 먼저 나온다. 하지만 실제로 더 중요한 질문은 "어떤 라이브러리를 쓸까"보다 "이 상태는 어디에 있어야 자연스러운가"다.

상태를 전부 전역 store에 넣으면 처음에는 편하다. 그런데 시간이 지나면 서버에서 다시 받아야 하는 데이터, URL에 남아야 하는 필터, 모달 열림 여부, 폼 입력값이 한곳에 섞인다.

이렇게 되면 문제가 생겼을 때 원인을 찾기 어렵다.

- 새로고침하면 유지되어야 하는 값인가?
- 뒤로 가기와 앞으로 가기에 반응해야 하는 값인가?
- 서버 데이터와 동기화되어야 하는 값인가?
- 화면 안에서 잠깐 쓰고 사라지는 값인가?
- 여러 탭이나 여러 사용자가 바꿀 수 있는 값인가?

이 질문에 따라 상태의 위치가 달라진다.

## 서버 상태는 내가 소유한 값이 아니다

서버 상태는 프론트엔드가 소유한 값이 아니다. 서버에 원본이 있고, 클라이언트는 그 스냅샷을 잠시 들고 있는 것이다.

예를 들어 사용자 목록, 주문 상태, 댓글, 알림 개수는 대부분 서버 상태다.

```ts
const { data, isLoading, isFetching, error } = useQuery({
  queryKey: ["orders", userId],
  queryFn: () => fetchOrders(userId),
  staleTime: 30_000
});
```

이런 데이터는 단순 전역 상태보다 React Query 같은 서버 상태 도구가 잘 맞는다. 이유는 데이터 값 자체보다 그 주변 문제가 더 크기 때문이다.

- 캐시
- refetch
- stale 여부
- retry
- loading과 fetching 구분
- mutation 이후 invalidation
- 중복 요청 제거

전역 store에 `orders` 배열만 넣으면 이런 정책을 직접 만들어야 한다. 서버 상태를 잘 다룬다는 것은 데이터를 저장하는 것이 아니라, 데이터의 신선도와 동기화 규칙을 같이 다루는 것이다.

## URL 상태는 공유와 복구가 필요한 값이다

검색어, 정렬, 페이지 번호, 탭, 필터처럼 화면의 위치를 설명하는 값은 URL에 두는 것이 자연스러울 때가 많다.

```txt
/products?keyword=keyboard&sort=price&page=2
```

이렇게 하면 장점이 많다.

- 새로고침해도 같은 화면이 복구된다.
- 링크를 공유할 수 있다.
- 뒤로 가기와 앞으로 가기가 자연스럽다.
- 서버 렌더링이나 초기 데이터 요청과 연결하기 쉽다.

반대로 이런 값을 컴포넌트 state나 전역 store에만 두면 사용자는 URL을 복사해도 같은 화면을 전달할 수 없다.

```tsx
function useProductSearchParams() {
  const searchParams = useSearchParams();

  return {
    keyword: searchParams.get("keyword") ?? "",
    sort: searchParams.get("sort") ?? "recent",
    page: Number(searchParams.get("page") ?? 1)
  };
}
```

URL 상태는 문자열 기반이라 파싱과 기본값 처리가 필요하다. 그래서 여러 곳에서 직접 읽기보다 작은 hook이나 parser로 감싸두면 좋다.

## 클라이언트 UI 상태는 가까운 곳에 둔다

모달 열림 여부, 드롭다운 상태, hover에 가까운 선택 상태, 임시 입력값처럼 화면 안에서만 의미가 있는 값은 가까운 컴포넌트에 두는 것이 좋다.

```tsx
function ProductFilterButton() {
  const [isOpen, setIsOpen] = useState(false);

  return (
    <>
      <button onClick={() => setIsOpen(true)}>필터</button>
      {isOpen ? <FilterModal onClose={() => setIsOpen(false)} /> : null}
    </>
  );
}
```

이 상태를 전역 store에 넣으면 다른 화면에서도 접근할 수는 있다. 하지만 접근 가능성이 항상 장점은 아니다. 필요 없는 곳에서도 바꿀 수 있고, 상태 변경 영향 범위가 넓어진다.

상태는 가능한 한 의미가 있는 가장 가까운 위치에 두고, 정말 여러 곳에서 필요할 때만 올리는 편이 좋다.

## optimistic update는 서버 상태와 UI 상태의 경계다

좋아요 버튼이나 즐겨찾기처럼 사용자가 즉시 반응을 기대하는 기능은 optimistic update를 쓰기도 한다.

```ts
const mutation = useMutation({
  mutationFn: toggleFavorite,
  onMutate: async (productId) => {
    await queryClient.cancelQueries({ queryKey: ["products"] });

    const previousProducts = queryClient.getQueryData<Product[]>(["products"]);

    queryClient.setQueryData<Product[]>(["products"], (products = []) =>
      products.map((product) =>
        product.id === productId
          ? { ...product, favorite: !product.favorite }
          : product
      )
    );

    return { previousProducts };
  },
  onError: (_error, _productId, context) => {
    queryClient.setQueryData(["products"], context?.previousProducts);
  },
  onSettled: () => {
    queryClient.invalidateQueries({ queryKey: ["products"] });
  }
});
```

여기서 중요한 점은 낙관적 업데이트가 "서버 상태를 클라이언트 상태로 바꾼다"는 뜻이 아니라는 것이다. 사용자의 체감 반응을 위해 캐시를 잠깐 먼저 바꾸고, 실패하면 되돌리고, 마지막에는 서버와 다시 맞춘다.

즉 optimistic update는 경계가 더 중요하다. 롤백할 이전 값, 실패 처리, 최종 invalidation이 같이 있어야 안정적이다.

## 폼 상태는 서버 상태와 분리한다

수정 화면에서 자주 하는 실수는 서버에서 받은 데이터를 그대로 폼 상태처럼 쓰는 것이다.

```tsx
const { data: profile } = useQuery({
  queryKey: ["profile"],
  queryFn: fetchProfile
});
```

서버에서 받은 `profile`은 원본 스냅샷이다. 사용자가 input에서 수정 중인 값은 임시 편집 상태다. 둘은 분리하는 편이 좋다.

```tsx
const form = useForm({
  defaultValues: {
    nickname: profile.nickname,
    bio: profile.bio
  }
});
```

사용자가 입력 중인 값은 아직 서버 상태가 아니다. 저장 버튼을 누르고 mutation이 성공한 뒤에야 서버 상태와 다시 연결된다.

이 경계를 흐리면 refetch가 일어났을 때 사용자가 입력 중인 값이 덮이거나, 저장하지 않은 값이 서버 데이터처럼 취급되는 문제가 생길 수 있다.

좋은 상태 관리는 한 도구로 모든 것을 해결하는 것이 아니다. 상태의 소유권과 수명을 보고 각자 어울리는 자리에 두는 것이다.

이 경계가 선명하면 컴포넌트가 단순해지고, refetch나 뒤로 가기, 새로고침 같은 실제 사용자 행동에도 화면이 덜 흔들린다. 결국 상태 관리는 라이브러리 실력이 아니라, 데이터가 어디에서 왔고 얼마나 오래 살아야 하는지 구분하는 설계 문제에 가깝다.
