## AI를 잘 쓴다는 것은 답을 빨리 받는 것이 아니었다

AI를 쓰면 검색, 초안 작성, 코드 수정, 문서 정리가 빨라진다. 하지만 그냥 "알아서 해줘"라고 맡기면 결과가 들쭉날쭉해진다. AI는 내가 머릿속으로 알고 있는 프로젝트 맥락, 사용자 의도, 팀의 기준을 자동으로 알지 못하기 때문이다.

AI를 생산성 도구로 쓰려면 질문을 잘하는 것보다 작업 흐름을 잘 설계해야 했다. 내가 원하는 결과를 정하고, AI가 참고할 근거를 주고, 나온 답을 실행해서 검증하는 흐름이 필요했다.

<Callout>
AI는 정답을 보장하는 기계가 아니라 빠르게 후보를 만드는 도구다. 생산성은 후보를 빨리 만드는 속도와, 틀린 후보를 빨리 걸러내는 검증 습관이 같이 있을 때 올라간다.
</Callout>

## AI는 대충 어떻게 답을 만드는가

LLM은 문장을 읽고 다음에 올 가능성이 높은 토큰을 예측하면서 답을 만든다. 여기서 토큰은 단어와 비슷하지만 완전히 같지는 않은 작은 텍스트 조각이다. 모델은 많은 코드와 문서, 대화 패턴을 학습했기 때문에 자연스러운 설명과 코드를 만들 수 있다.

중요한 점은 AI가 "진짜로 이해했다"기보다, 주어진 입력과 학습된 패턴을 바탕으로 그럴듯한 답을 만든다는 것이다. 그래서 입력에 정보가 부족하면 빈칸을 추측으로 채운다. 최신 API, 우리 프로젝트만의 규칙, 숨겨진 버그 재현 조건처럼 입력에 없는 정보는 틀릴 수 있다.

그래서 좋은 AI 사용은 다음 질문에서 시작한다.

- AI가 반드시 알아야 하는 정보는 무엇인가
- AI가 추측하면 위험한 부분은 무엇인가
- 답이 맞는지 어떻게 확인할 것인가
- 사람이 최종 판단해야 하는 기준은 무엇인가

## Context는 AI에게 주는 작업 재료다

Context는 이번 작업을 해결하는 데 필요한 주변 정보다. 코드 작업이라면 관련 파일, 에러 로그, 재현 방법, 기존 패턴, 테스트 명령이 context가 된다. 글쓰기라면 독자, 글의 목적, 포함할 개념, 말투, 참고한 자료가 context가 된다.

Context를 많이 주는 것이 항상 좋은 것은 아니다. 관련 없는 파일과 긴 설명을 많이 넣으면 중요한 신호가 흐려진다. 좋은 context는 짧고 정확해야 한다.

```txt
목표:
AI 섹션에 RAG와 생산성 활용법을 설명하는 글을 추가한다.

Context:
- content/ai의 기존 글 톤
- src/app/ai/page.tsx의 카드 구조
- src/app/globals.css의 색상 토큰과 반응형 규칙

제약:
- 새 라이브러리는 설치하지 않는다.
- 기존 디자인 톤을 유지한다.
- 모바일에서 카드 텍스트가 넘치지 않게 한다.

완료 기준:
- npm run build가 통과한다.
- /ai 목록과 상세 글이 정상 렌더링된다.
```

이런 식으로 주면 AI는 "무엇을", "어디에", "어떤 기준으로" 해야 하는지 알 수 있다.

## RAG는 AI에게 근거를 검색하게 만드는 방식이다

사용자가 말한 REG는 보통 AI 문맥에서는 RAG를 뜻하는 경우가 많다. RAG는 Retrieval-Augmented Generation의 약자다. 한국어로 풀면 "검색으로 보강한 생성"에 가깝다.

일반적인 AI 답변은 모델이 이미 알고 있는 패턴에 의존한다. RAG는 답을 만들기 전에 먼저 관련 문서나 데이터 조각을 검색하고, 그 검색 결과를 context로 넣은 뒤 답을 만들게 한다.

흐름은 단순하다.

```txt
사용자 질문
-> 관련 문서 검색
-> 필요한 문서 조각 선택
-> 선택된 근거를 AI에게 전달
-> 근거를 바탕으로 답변 생성
```

예를 들어 회사 내부 디자인 시스템 문서가 있다고 하자. AI가 기억만으로 버튼 컴포넌트를 만들면 팀 규칙과 다르게 만들 수 있다. 하지만 RAG로 디자인 시스템 문서에서 버튼 규칙을 찾아 context에 넣으면, AI는 그 근거를 바탕으로 더 정확한 코드를 만들 수 있다.

RAG가 특히 좋은 상황은 다음과 같다.

- 사내 문서, API 문서, 디자인 시스템처럼 모델이 원래 모르는 자료를 써야 할 때
- 최신 공식 문서 기준으로 답해야 할 때
- 긴 문서 묶음에서 관련 부분만 찾아 요약해야 할 때
- 코드베이스에서 유사한 구현을 찾아 같은 패턴으로 수정해야 할 때

반대로 간단한 문장 다듬기, 이미 알고 있는 코드 패턴의 초안, 작은 리팩터링에는 RAG까지 필요하지 않을 수 있다. 도구는 문제 크기에 맞게 쓰는 편이 좋다.

## 생산성을 올리는 사용 순서

AI를 잘 쓰는 순서는 복잡하지 않았다.

1. 먼저 목표를 한 문장으로 쓴다.
2. AI가 봐야 할 파일과 자료를 고른다.
3. 바꾸면 안 되는 제약을 적는다.
4. AI에게 초안을 만들게 한다.
5. diff, 빌드, 테스트, 화면으로 검증한다.
6. 반복되는 흐름은 AGENTS.md, MCP, Skill로 옮긴다.

처음부터 자동화를 크게 만들 필요는 없다. 같은 설명을 세 번 반복하게 되면 그때 문서화하면 된다. 같은 외부 자료를 계속 복사해서 붙여넣고 있다면 MCP나 RAG 연결을 고민하면 된다. 같은 리뷰 기준을 계속 말하고 있다면 skill로 만들면 된다.

## 개발자가 AI에게 맡기기 좋은 일

AI는 빠른 탐색과 초안 작성에 강하다. 특히 프론트엔드 개발에서는 다음 작업에 잘 맞았다.

- 기존 컴포넌트 패턴 찾기
- 반복되는 JSX와 스타일 정리
- 타입 에러 원인 후보 찾기
- 테스트 케이스 초안 만들기
- 에러 로그를 재현 조건으로 정리하기
- 기술 글의 구조와 예시 다듬기
- 공식 문서를 읽고 프로젝트 코드에 적용할 부분 요약하기

하지만 제품 방향, 사용자 경험의 우선순위, 민감한 데이터 처리, 배포 판단은 사람이 잡아야 한다. AI가 빠르게 만든 코드는 "후보"이지 "결정"이 아니다.

## 좋은 프롬프트는 짧은 작업 명세서다

프롬프트를 길게 쓰는 것보다 중요한 것은 빠진 기준이 없게 쓰는 것이다.

```txt
Goal:
/ai 페이지에 RAG와 AI 생산성 활용법을 설명하는 섹션을 추가해줘.

Context:
- 기존 AI 글은 content/ai에 MDX로 있다.
- 디자인은 src/app/globals.css의 card, section-block, pill 스타일을 따른다.

Constraints:
- 새 dependency 추가 금지.
- 기존 글 삭제 금지.
- 모바일에서 1열로 자연스럽게 내려가야 함.

Done when:
- 새 글이 /ai 목록에 보인다.
- 상세 페이지에서 MDX가 깨지지 않는다.
- npm run build 통과.
```

이 구조는 거의 모든 개발 작업에 재사용할 수 있다. 핵심은 AI에게 자유를 주되, 위험한 추측은 줄이는 것이다.

## 검증 루프가 없으면 생산성이 아니라 부채가 된다

AI 답변을 그대로 믿으면 처음에는 빨라 보일 수 있다. 하지만 틀린 코드가 들어가면 나중에 디버깅 비용이 더 커진다. 그래서 AI를 쓸수록 검증 루프가 중요해진다.

내가 보는 순서는 이렇다.

- 변경된 파일이 요청 범위 안에 있는지 확인한다.
- 기존 패턴과 다른 추상화가 생기지 않았는지 본다.
- 타입, 빌드, 테스트를 실행한다.
- 화면이 있는 작업은 실제 브라우저나 시뮬레이터에서 본다.
- 이해하지 못한 코드는 다시 설명시키고 직접 읽는다.

이 과정이 있어야 AI가 만든 속도가 실제 생산성으로 남는다.

## 결론

AI를 잘 쓰는 핵심은 "더 똑똑한 질문" 하나가 아니었다. 목표를 정하고, context를 고르고, 필요하면 RAG나 MCP로 근거를 가져오고, 반복 작업은 skill로 저장하고, 마지막에는 사람이 검증하는 흐름이었다.

AI는 개발자를 대체하는 버튼이라기보다 작업 속도를 높이는 증폭기였다. 방향이 틀리면 실수도 빠르게 커지고, 방향이 맞으면 학습과 구현 속도가 같이 올라간다. 그래서 AI 시대의 생산성은 도구 사용법보다 작업을 구조화하는 능력에 더 가까웠다.
