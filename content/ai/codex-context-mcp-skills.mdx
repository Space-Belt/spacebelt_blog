## Codex를 잘 쓴다는 것은 질문을 잘 설계한다는 뜻이었다

Codex를 잘 쓰는 방법은 단순히 "코드를 만들어줘"라고 말하는 것이 아니었다. 내가 원하는 결과, 코드베이스의 맥락, 지켜야 할 제약, 완료 기준을 정확히 전달하는 일이었다.

개발 AI는 빠르게 코드를 만들 수 있지만, 프로젝트의 의도와 사용자의 경험을 처음부터 알고 있지는 않다. 그래서 개발자가 해야 할 일은 AI에게 모든 것을 맡기는 것이 아니라, AI가 올바른 방향으로 일할 수 있도록 작업 환경을 설계하는 것이었다.

<Callout>
AI 활용 능력은 프롬프트를 길게 쓰는 능력이 아니라, 필요한 맥락을 고르고, 외부 도구를 연결하고, 반복되는 작업을 재사용 가능한 흐름으로 바꾸는 능력이었다.
</Callout>

## Context는 많이 주는 것이 아니라 정확히 주는 것이었다

Context는 AI가 작업을 이해하기 위해 필요한 주변 정보였다. 하지만 context를 많이 넣는다고 항상 좋은 결과가 나오는 것은 아니었다. 필요 없는 파일, 오래된 설명, 관련 없는 요구사항을 많이 넣으면 AI는 오히려 중요한 신호를 놓칠 수 있었다.

비유하면 context는 요리사에게 주는 재료와 레시피였다. 냉장고 전체를 통째로 넘기는 것보다, 오늘 만들 요리에 필요한 재료와 맛의 기준을 정확히 알려주는 편이 더 좋은 결과를 만들었다.

Codex에게 작업을 줄 때 내가 기준으로 삼는 context는 네 가지였다.

- 목표: 무엇을 만들거나 고치려는지
- 관련 파일: 어떤 파일과 폴더를 봐야 하는지
- 제약: 기존 패턴, 디자인 규칙, 금지해야 할 변경은 무엇인지
- 완료 기준: 어떤 테스트나 화면 확인이 끝나야 완료인지

예를 들어 검색 기능을 추가한다면 이렇게 요청하는 편이 좋았다.

```txt
목표:
블로그, AI 글, 프로젝트를 한 번에 찾는 검색 기능을 추가한다.

관련 파일:
- src/lib/content.ts
- src/components/site-header.tsx
- src/app/page.tsx
- src/app/globals.css

제약:
- 새 검색 라이브러리는 설치하지 않는다.
- 기존 카드 스타일과 색상 토큰을 유지한다.
- 검색 결과는 제목, 설명, 태그까지만 보여준다.

완료 기준:
- npm run build가 통과한다.
- /, /blog, /ai에서 검색 버튼이 정상 동작한다.
```

이 요청은 길지만 복잡하지 않았다. Codex가 봐야 할 범위와 바꾸면 안 되는 기준이 분명했다. 반대로 "검색 기능 만들어줘"라고만 하면, AI는 검색 범위, UI 위치, 상태 관리 방식, 라이브러리 사용 여부를 모두 추측해야 했다.

## AGENTS.md는 매번 반복하는 설명을 줄이는 장치였다

매번 같은 규칙을 프롬프트에 쓰는 것은 비효율적이었다. 예를 들어 "TypeScript를 사용해줘", "빌드 확인해줘", "기존 스타일을 따라줘", "사용자 변경을 되돌리지 마" 같은 규칙은 거의 모든 작업에서 반복된다.

이런 규칙은 `AGENTS.md`에 넣는 것이 좋았다. `AGENTS.md`는 AI가 프로젝트에 들어왔을 때 먼저 읽는 작업 안내서에 가까웠다.

```md
# AGENTS.md

## Project Rules

- Use TypeScript for new code.
- Follow existing App Router conventions.
- Do not add production dependencies without asking.
- Keep UI components responsive on mobile.
- Run `npm run build` after route or MDX changes.
- Do not rewrite unrelated files.
```

비유하면 `AGENTS.md`는 신입 개발자에게 주는 온보딩 문서였다. 매번 말로 설명하지 않아도, 프로젝트에 들어오면 먼저 읽고 일하는 기준이었다.

좋은 `AGENTS.md`는 길지 않아야 했다. 너무 많은 철학과 추상적인 문장을 넣으면 AI가 실제 작업에서 무엇을 해야 하는지 판단하기 어려웠다. 실행 가능한 규칙, 명령어, 금지 사항, 완료 기준을 짧게 적는 편이 좋았다.

## MCP는 외부 세계와 연결하는 통로였다

MCP는 Model Context Protocol의 약자였다. Codex가 로컬 코드만 보는 것이 아니라, 외부 문서나 도구, 디자인 자료, 브라우저, 에러 로그 같은 정보를 가져올 수 있게 하는 연결 방식이었다.

비유하면 MCP는 작업실에 연결된 전용 전화선이었다. Codex가 혼자 기억에 의존해서 추측하는 대신, 필요한 곳에 직접 물어보고 최신 정보를 가져올 수 있게 해주었다.

프론트엔드 개발에서 MCP가 특히 유용한 상황은 다음과 같았다.

- 공식 문서가 자주 바뀌는 라이브러리를 확인해야 할 때
- Figma 디자인을 보고 실제 UI를 구현해야 할 때
- 브라우저 화면을 열어 레이아웃이나 콘솔 에러를 확인해야 할 때
- Sentry 같은 도구에서 실제 사용자 에러 로그를 읽어야 할 때
- 회사 내부 API 문서나 디자인 시스템 문서를 반복해서 참고해야 할 때

MCP를 연결하는 예시는 이런 형태였다. 예를 들어 OpenAI 개발자 문서를 Codex에서 바로 검색하고 읽고 싶다면 문서 MCP를 추가할 수 있었다.

```bash
codex mcp add openaiDeveloperDocs --url https://developers.openai.com/mcp
```

또는 `config.toml`에 직접 설정할 수도 있었다.

```toml
[mcp_servers.openaiDeveloperDocs]
url = "https://developers.openai.com/mcp"
```

MCP를 많이 연결한다고 좋은 환경이 되는 것은 아니었다. 필요한 도구만 연결해야 했다. 모든 도구를 한 번에 붙이면 AI가 사용할 수 있는 선택지는 많아지지만, 실제 작업은 느려지고 판단도 흐려질 수 있었다.

내 기준은 단순했다. 자주 복사해서 붙여넣는 외부 정보가 있다면 MCP 후보였다. 가끔 한 번 보는 자료라면 그냥 프롬프트에 링크나 요약을 주는 것으로 충분했다.

## Skills는 반복 작업을 재사용 가능한 레시피로 만드는 방법이었다

Skills는 Codex에게 특정 작업을 더 안정적으로 하게 만드는 작업 레시피였다. 단순한 프롬프트 모음이 아니라, 언제 사용해야 하는지, 어떤 순서로 작업해야 하는지, 어떤 자료와 스크립트를 참고해야 하는지 묶어둘 수 있었다.

비유하면 skill은 요리책의 한 페이지였다. 매번 "재료를 씻고, 썰고, 볶고, 간을 맞춰줘"라고 말하지 않아도, "이 레시피로 해줘"라고 말하면 같은 순서로 일하게 만드는 장치였다.

기본 구조는 이렇게 잡을 수 있었다.

```txt
.agents/skills/
  blog-writer/
    SKILL.md
    references/
      writing-style.md
      article-template.md
```

`SKILL.md`는 이렇게 작성할 수 있었다.

```md
---
name: "blog-writer"
description: "Use when turning raw study notes into portfolio-ready MDX blog posts for this Next.js site."
---

# Blog Writer

1. Read the user's raw notes.
2. Choose a clear title and description.
3. Keep the tone educational and practical.
4. Use MDX frontmatter required by `content/blog`.
5. Include examples only when they clarify the concept.
6. End with a short personal learning summary.
```

이 skill이 있으면 매번 블로그 글 형식과 말투를 길게 설명하지 않아도 된다. "이 메모를 `$blog-writer`로 정리해줘"라고 요청하면, Codex가 정해둔 흐름을 따라 글을 만들 수 있다.

## Context, MCP, Skills는 역할이 달랐다

세 가지는 비슷해 보이지만 역할이 달랐다.

Context는 이번 작업에 필요한 정보였다. 예를 들어 "이 파일을 보고 이 버그를 고쳐줘"에서 파일, 에러 로그, 원하는 결과가 context였다.

MCP는 외부 정보를 가져오는 통로였다. 예를 들어 최신 공식 문서, Figma 디자인, 브라우저 상태, 에러 추적 도구에 접근해야 할 때 MCP를 사용했다.

Skills는 반복되는 작업 방식을 저장하는 레시피였다. 예를 들어 블로그 글 작성, PR 리뷰, 릴리즈 노트 작성, 접근성 점검처럼 반복되는 절차가 있으면 skill로 만들 수 있었다.

정리하면 다음과 같았다.

```txt
Context = 이번 작업에 필요한 재료
MCP = 외부 재료를 가져오는 통로
Skills = 반복 작업을 처리하는 레시피
AGENTS.md = 프로젝트 전체의 작업 규칙
```

## 예시 1: React Native 버그를 Codex와 함께 고치는 흐름

React Native에서 상품 상세 화면의 옵션 선택 버그를 고친다고 가정해보았다. 좋은 요청은 이렇게 시작할 수 있었다.

```txt
목표:
상품 상세 화면에서 옵션을 바꿨을 때 가격 표시가 늦게 바뀌는 문제를 고친다.

Context:
- app/screens/ProductDetailScreen.tsx
- app/components/OptionSelector.tsx
- app/stores/cartDraftStore.ts
- 재현 방법: 옵션 A 선택 후 옵션 B를 빠르게 누르면 이전 가격이 잠깐 보인다.

제약:
- Zustand store 구조는 크게 바꾸지 않는다.
- 화면 전환 로직은 건드리지 않는다.
- 터치 영역과 선택 상태 스타일은 유지한다.

완료 기준:
- 가격 표시가 선택 직후 일관되게 바뀐다.
- 관련 unit test를 추가하거나 기존 테스트를 갱신한다.
```

여기서 context는 관련 파일과 재현 방법이었다. MCP가 필요하다면 React Navigation 문서나 디자인 시스템 문서를 연결할 수 있었다. 같은 유형의 모바일 버그를 반복해서 고친다면 `rn-debugging` skill을 만들 수 있었다.

## 예시 2: Figma 기반 웹 UI를 구현하는 흐름

포트폴리오 메인 화면을 디자인에서 구현한다고 가정하면 MCP가 더 중요해졌다. 디자인 정보는 repo 안에 없기 때문이었다.

```txt
목표:
Figma 디자인을 기준으로 포트폴리오 메인 hero와 프로젝트 섹션을 구현한다.

MCP:
- Figma MCP로 frame 이름 `Portfolio / Home`을 확인한다.

Context:
- src/app/page.tsx
- src/app/globals.css
- 기존 ProjectCard 컴포넌트

제약:
- 메인 canvas나 3D 씬은 카드 안에 넣지 않는다.
- 모바일에서 텍스트가 넘치지 않아야 한다.
- 새 색상은 기존 CSS 변수 안에서만 추가한다.

완료 기준:
- 데스크톱과 모바일에서 레이아웃이 깨지지 않는다.
- npm run build가 통과한다.
```

이 흐름에서 MCP는 디자인이라는 외부 정보를 가져오는 역할이었다. Codex는 디자인을 보고 구현할 수 있지만, 구현 기준과 완료 기준은 개발자가 정해야 했다.

## 예시 3: 블로그 글 작성 skill을 만드는 흐름

이 포트폴리오에서는 공부한 내용을 계속 글로 남기는 것이 중요했다. 매번 글의 톤, frontmatter, 섹션 구성, 예시 수준을 설명하는 대신 skill로 만들 수 있었다.

```md
---
name: "portfolio-ai-writer"
description: "Use when writing AI or frontend study notes as MDX posts for the portfolio."
---

# Portfolio AI Writer

## Output

- Create one `.mdx` file under `content/blog` or `content/ai`.
- Use frontmatter with title, description, date, tags.
- Explain concepts with one practical analogy.
- Include one realistic frontend example.
- Keep claims based on what the user actually studied or built.
- Run `npm run build` after writing.
```

이 skill은 단순히 글을 예쁘게 쓰기 위한 것이 아니었다. 글이 항상 같은 품질 기준을 통과하게 만드는 장치였다. 채용자가 봤을 때 글마다 구조와 톤이 안정적이면, 공부 기록도 더 신뢰감 있게 보였다.

## 예시 4: PR 리뷰를 자동화하는 흐름

반복되는 코드 리뷰도 skill로 만들기 좋았다.

```md
---
name: "frontend-review"
description: "Use when reviewing React, Next.js, or React Native changes before merging."
---

# Frontend Review

Check in this order:

1. User-facing behavior regression
2. State and data flow correctness
3. Accessibility and mobile layout
4. Performance risks
5. Missing tests or missing verification

Always report findings first with file references.
```

이렇게 해두면 "리뷰해줘"라고만 말해도 Codex가 프로젝트의 리뷰 기준을 따를 수 있었다. 특히 프론트엔드에서는 UI가 보이는 것만으로는 충분하지 않았다. 상태 흐름, 접근성, 모바일 레이아웃, 렌더링 비용까지 봐야 했다.

## 내가 생각하는 좋은 사용 순서

처음부터 MCP와 skills를 모두 세팅하려고 하면 오히려 복잡해질 수 있었다. 좋은 순서는 작게 시작하는 것이었다.

1. 먼저 프롬프트에 goal, context, constraints, done when을 넣는 습관을 만든다.
2. 반복해서 말하는 규칙은 `AGENTS.md`에 옮긴다.
3. 자주 복사해서 붙여넣는 외부 자료가 있으면 MCP를 연결한다.
4. 같은 작업을 세 번 이상 반복하면 skill로 만든다.
5. AI가 같은 실수를 반복하면 회고를 시키고 `AGENTS.md`나 skill을 업데이트한다.

이 순서가 좋은 이유는 간단했다. 먼저 내가 실제로 어떤 작업을 반복하는지 알아야 했다. 반복 패턴을 모른 채 skill부터 만들면, 사용하지 않는 문서만 늘어날 수 있었다.

## 나쁜 사용법도 분명히 있었다

Codex를 잘 쓰려면 피해야 할 습관도 있었다.

- 에러 로그만 던지고 "고쳐줘"라고 말하는 것
- 최신 문서가 필요한데 AI의 기억에만 의존하는 것
- 모든 MCP 서버를 한 번에 연결하는 것
- 작은 반복 작업도 매번 긴 프롬프트로 처리하는 것
- AI가 만든 코드를 읽지 않고 바로 합치는 것
- 빌드와 테스트 없이 완료로 보는 것

AI를 쓰면 속도는 빨라질 수 있다. 하지만 검증 없이 빨라지는 것은 위험했다. 특히 프론트엔드에서는 화면은 그럴듯해도 상태 흐름, 접근성, 모바일 레이아웃, 성능 문제가 숨어 있을 수 있었다.

## 결론

Codex를 잘 사용하는 핵심은 AI에게 일을 던지는 것이 아니라, AI가 잘 일할 수 있는 환경을 만드는 것이었다.

Context는 이번 작업의 재료를 고르는 일이었다. MCP는 repo 밖의 살아 있는 정보를 연결하는 일이었다. Skills는 반복되는 작업 방식을 레시피로 만드는 일이었다. `AGENTS.md`는 프로젝트 전체의 기본 규칙을 자동으로 전달하는 일이었다.

이 네 가지를 구분해서 사용할 수 있으면 AI 활용은 단순한 생산성 도구를 넘어 개발 프로세스의 일부가 될 수 있었다. React와 React Native 개발에서도 마찬가지였다. 화면을 구현하는 속도보다 중요한 것은, 요구사항을 정확히 이해하고, 근거를 확인하고, 변경 결과를 검증하는 흐름이었다.
