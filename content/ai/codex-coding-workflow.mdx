## Codex를 잘 쓰는 기준

Codex를 잘 쓴다는 것은 코드를 대신 쓰게 만드는 것이 아니었다. 내가 해결하려는 문제, 프로젝트의 제약, 건드려도 되는 파일, 확인해야 할 테스트를 명확히 주고 결과를 검증하는 일이었다.

AI에게 일을 맡길 때 가장 위험한 순간은 "대충 알아서 해줘"라고 말하는 순간이었다. 개발자는 맥락을 알고 있지만, AI는 내가 알려준 맥락과 코드베이스에서 읽은 정보로만 판단한다. 그래서 좋은 요청은 작은 작업 명세서에 가까웠다.

<Callout>
Codex는 빠른 손과 넓은 탐색 능력을 가진 도구다. 하지만 제품의 의도, 변경의 우선순위, 위험 판단은 개발자가 해야 했다.
</Callout>

## 좋은 요청에는 경계가 있었다

좋은 요청은 무엇을 만들지뿐 아니라, 어디까지 바꿔도 되는지 알려주었다.

```txt
블로그 검색 기능을 추가해줘.

조건:
- content/blog의 MDX frontmatter를 검색 대상으로 사용한다.
- 헤더에는 작은 검색 버튼을 둔다.
- 검색 결과는 제목, 설명, 태그를 보여준다.
- 기존 프로젝트 카드 스타일은 유지한다.
- 빌드가 통과하는지 확인한다.
```

이런 요청은 AI가 불필요하게 넓은 범위를 건드리지 않게 만든다. 반대로 "검색 기능 만들어줘"라고만 말하면, 새로운 라이브러리를 설치하거나, 필요 이상의 상태 관리 구조를 만들거나, 기존 디자인과 다른 화면을 만들 가능성이 커진다.

## 코드를 맡긴 뒤에는 diff를 읽어야 했다

Codex가 코드를 작성해도 최종 책임은 개발자에게 있었다. 그래서 결과를 받을 때는 완성 화면만 보는 것이 아니라 diff를 읽어야 했다.

내가 확인하는 순서는 다음과 같았다.

- 요청한 파일만 바뀌었는지 확인한다.
- 기존 구조와 맞는 방식으로 구현되었는지 본다.
- 새로 생긴 상태와 props가 필요한지 본다.
- 빌드와 타입 체크가 통과하는지 확인한다.
- 사용자 흐름이 끊기지 않는지 브라우저에서 확인한다.

AI가 만든 코드는 초안이었다. 초안을 읽고 프로젝트의 언어로 다시 다듬는 과정이 개발자의 일이었다.

## Codex에게 좋은 작업

Codex는 코드베이스를 읽고 변경하는 작업에 강했다. 특히 반복되는 패턴을 찾아 적용하거나, 여러 파일을 일관되게 수정하는 데 유용했다.

예를 들면 다음 작업이 잘 맞았다.

- 컴포넌트 구조 정리
- 타입 에러 수정
- 라우트 추가
- MDX 콘텐츠 템플릿 생성
- 테스트 케이스 초안 작성
- 작은 리팩터링
- 스타일 누락 정리

하지만 제품 방향, 기능 우선순위, 사용자 경험의 기준은 AI보다 개발자가 더 잘 알아야 했다. AI가 제안한 구현이 맞아 보여도, 실제 사용자 흐름과 맞지 않으면 바꾸어야 했다.

## 프론트엔드 개발에서의 활용

React와 React Native 작업에서는 AI에게 한 번에 큰 앱을 맡기기보다 화면 단위로 나누는 것이 좋았다.

```txt
상품 상세 화면의 옵션 선택 영역만 개선해줘.

현재 문제:
- 옵션을 바꿀 때 가격이 늦게 반영된다.
- 선택된 옵션의 시각적 상태가 약하다.
- 모바일에서 터치 영역이 좁다.

원하는 결과:
- 상태 계산은 읽기 쉽게 분리한다.
- 터치 영역은 최소 44px 이상으로 둔다.
- 기존 색상 토큰을 유지한다.
```

이렇게 요청하면 AI가 화면 전체를 다시 만드는 대신, 실제 문제가 있는 부분에 집중할 수 있었다.

## 테스트는 AI 답변의 마침표였다

AI가 작성한 코드는 실행 전까지 믿을 수 없었다. 빌드, 타입 체크, 테스트, 브라우저 확인은 반드시 필요했다.

```bash
npm run build
```

테스트가 실패하면 "왜 실패했는지"를 다시 AI에게 물어볼 수 있었다. 하지만 이때도 에러 로그 전체를 던지는 것보다, 어떤 명령을 실행했고 어떤 파일을 바꾸었고 어떤 에러가 났는지 정리해서 주는 편이 좋았다.

## 내가 지키는 원칙

Codex를 사용할 때 지키는 원칙은 단순했다.

- 비밀키와 민감한 정보는 주지 않는다.
- 작업 범위를 작게 나눈다.
- 변경된 코드를 직접 읽는다.
- 테스트로 확인한다.
- 이해하지 못한 코드는 그대로 합치지 않는다.

AI를 잘 쓰는 개발자는 AI에게 모든 것을 맡기는 사람이 아니었다. AI가 빠르게 만든 후보를 읽고, 검증하고, 제품에 맞게 선택하는 사람이었다.
