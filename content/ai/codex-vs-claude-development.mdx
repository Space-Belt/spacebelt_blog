## 둘 다 코딩 에이전트지만 감각은 달랐다

Codex와 Claude Code는 둘 다 개발 AI였다. 더 정확히 말하면 단순히 질문에 답하는 챗봇이 아니라, 코드베이스를 읽고, 파일을 수정하고, 명령어를 실행하고, 결과를 확인하는 agentic coding tool에 가까웠다.

그래서 비교 기준도 "누가 설명을 더 잘하나"가 아니었다. 실제 개발 흐름에서 어떤 도구가 어떤 순간에 더 편했는지, 어떤 방식으로 요청해야 결과가 안정적인지, 어떤 검증 습관이 필요한지를 보는 것이 중요했다.

비유하면 둘 다 같은 작업실에 들어와서 같이 코딩할 수 있는 동료였다. 다만 Codex는 현재 작업의 목적과 파일 변경을 좁게 잡고 끝까지 밀어붙이는 느낌이 강했고, Claude Code는 터미널 중심의 긴 세션에서 코드 탐색, 계획, 명령 실행, 반복 작업 자동화를 넓게 이어가는 느낌이 강했다.

<Callout>
중요한 것은 어떤 도구가 무조건 더 좋다는 결론이 아니었다. 지금 필요한 일이 탐색인지, 구현인지, 검증인지, 자동화인지 구분하고 도구의 장점을 맞춰 쓰는 것이었다.
</Callout>

## Codex는 "요청한 변경을 안전하게 끝내는 흐름"에 잘 맞았다

Codex를 사용할 때 가장 좋았던 점은 작업 범위가 비교적 선명할 때였다. 예를 들어 Next.js 프로젝트에 검색 기능을 추가하거나, MDX 글 상세 페이지의 라우팅 문제를 고치거나, 컴포넌트 스타일을 기존 패턴에 맞게 정리하는 작업에 잘 맞았다.

```txt
목표:
블로그, AI 글, 프로젝트를 한 번에 검색하는 기능을 추가한다.

관련 파일:
- src/lib/content.ts
- src/components/site-header.tsx
- src/app/page.tsx
- src/app/globals.css

제약:
- 새 검색 라이브러리는 추가하지 않는다.
- 기존 디자인 톤을 유지한다.
- 모바일에서 헤더가 넘치지 않게 한다.

완료 기준:
- npm run build가 통과한다.
- /, /blog, /ai에서 검색 흐름이 깨지지 않는다.
```

이런 식으로 goal, context, constraints, done when을 주면 Codex는 작업을 좁혀서 처리하기 좋았다. 파일을 직접 수정하고, 빌드를 돌리고, 실패하면 다시 고치는 흐름이 자연스러웠다.

React와 React Native 작업에서도 마찬가지였다. 옵션 선택 버그, 상태 업데이트 문제, 타입 에러, 라우트 추가, 컴포넌트 분리처럼 실제 코드 변경이 필요한 일은 Codex에게 잘 맞았다.

## Claude Code는 "터미널 안에서 긴 개발 세션을 이어가는 흐름"에 강했다

Claude Code도 단순 설명용 Claude가 아니었다. 터미널, IDE, 데스크톱, 웹에서 코드베이스를 읽고 파일을 수정하고 명령어를 실행할 수 있는 개발 도구였다. 그래서 실제로는 Codex와 같은 범주의 코딩 에이전트로 비교하는 것이 맞았다.

Claude Code의 장점은 긴 세션에서 코드베이스를 탐색하고, 계획을 세우고, 작업을 단계별로 이어가는 흐름이었다. 특히 처음 보는 코드베이스를 파악하거나, 리팩터링 전에 구조를 이해하거나, 테스트와 리뷰 흐름을 반복할 때 유용했다.

```txt
이 코드베이스를 먼저 파악해줘.

보고 싶은 것:
- 주요 라우트 구조
- 상태 관리 방식
- API 요청이 모여 있는 위치
- 공통 UI 컴포넌트 패턴
- 리팩터링하면 위험할 수 있는 지점
```

이런 요청은 바로 파일을 고치기보다 읽기와 계획이 먼저 필요한 상황에 잘 맞았다. Claude Code는 broad question에서 시작해 specific question으로 좁혀가는 코드베이스 탐색 흐름에 강했다.

## Claude Code의 Plan Mode는 큰 변경 전에 좋았다

Claude Code에서 인상적인 부분은 Plan Mode였다. 복잡한 작업을 바로 수정하기보다, 먼저 read-only에 가까운 분석 흐름으로 요구사항을 확인하고 계획을 세우는 방식이었다.

예를 들어 React Native 앱의 인증 구조를 바꾸는 일은 바로 수정하기 위험했다. 화면 흐름, 토큰 저장, API 요청, 에러 처리, 네비게이션 가드가 함께 얽혀 있을 수 있기 때문이었다.

```txt
Plan Mode로 먼저 분석해줘.

목표:
React Native 앱의 로그인 유지 흐름을 정리한다.

확인할 것:
- 토큰 저장 위치
- 앱 시작 시 인증 상태 확인 흐름
- 로그인 실패와 만료 처리
- navigation guard 위치
- 바꾸면 위험한 파일

아직 코드는 수정하지 말고 계획만 작성해줘.
```

이런 작업에서는 Claude Code의 계획 중심 흐름이 좋았다. 개발자는 계획을 보고 위험한 부분을 먼저 확인한 뒤, 실제 수정으로 넘어갈 수 있었다.

## Claude Code는 CLAUDE.md와 memory 흐름이 중요했다

Claude Code를 제대로 쓰려면 `CLAUDE.md`를 잘 쓰는 것이 중요했다. `CLAUDE.md`는 프로젝트 규칙, 자주 쓰는 명령어, 아키텍처 기준, 리뷰 체크리스트를 저장하는 문서였다. Claude Code는 세션을 시작할 때 이런 정보를 context로 읽는다.

```md
# CLAUDE.md

## Project Rules

- Use TypeScript for new files.
- Follow existing App Router patterns.
- Run `npm run build` after route or MDX changes.
- Do not add dependencies without asking.
- Keep mobile layout readable under 390px width.
```

Codex에서는 `AGENTS.md`가 비슷한 역할을 했다. 두 도구 모두 매번 반복해서 말하는 규칙을 프로젝트 문서로 옮길 수 있었다. 차이는 파일 이름과 생태계의 세부 기능이었다.

비유하면 `CLAUDE.md`와 `AGENTS.md`는 AI에게 주는 팀 온보딩 문서였다. 매번 "우리 팀은 이렇게 일해"라고 설명하지 않아도, 프로젝트에 들어오면 먼저 읽고 작업 기준을 맞추는 문서였다.

## 파일 참조 방식도 실제 사용감에 영향을 주었다

Claude Code는 `@`를 이용해 파일과 디렉터리를 빠르게 참조하는 흐름이 명확했다.

```txt
@src/app/page.tsx 와 @src/app/globals.css 를 보고
홈 화면 hero가 모바일에서 어색한 이유를 설명해줘.
아직 수정하지 말고 원인과 수정 방향만 정리해줘.
```

이 방식은 "이 파일을 봐야 한다"는 context를 명확히 전달하기 좋았다. Codex에서도 파일 경로나 작업 범위를 명시하는 것이 중요했다. 결국 두 도구 모두 context 품질이 결과 품질을 크게 좌우했다.

## 둘 다 MCP를 붙이면 외부 맥락까지 가져올 수 있었다

Codex와 Claude Code 모두 MCP를 활용할 수 있었다. MCP는 repo 밖의 정보를 가져오는 연결 통로였다. 예를 들어 공식 문서, Figma, Jira, Slack, GitHub 이슈처럼 코드베이스 밖에 있는 정보가 필요할 때 유용했다.

프론트엔드 개발에서는 특히 Figma와 공식 문서 쪽 MCP가 쓸모 있었다. 디자인을 보고 구현하거나, 최신 라이브러리 API를 확인해야 하는 작업은 AI의 기억만 믿으면 위험했다.

```txt
Figma MCP로 Home 화면 디자인을 확인하고,
현재 src/app/page.tsx와 비교해서 빠진 UI 상태를 정리해줘.
아직 코드는 수정하지 말고 차이만 알려줘.
```

이런 요청은 AI를 단순 코드 생성기가 아니라 외부 맥락을 읽는 개발 보조 도구로 쓰는 방식이었다.

## Codex의 장점

내 기준에서 Codex의 장점은 다음과 같았다.

- 요청한 변경 범위를 좁게 잡고 구현하기 좋았다.
- 파일 수정, 패치, 빌드 확인 흐름이 단순했다.
- 기존 코드 스타일을 따라가며 변경하기 좋았다.
- 포트폴리오처럼 Next.js 파일 구조가 명확한 프로젝트에 잘 맞았다.
- "이거 만들어줘, 빌드까지 확인해줘" 형태의 작업이 자연스러웠다.
- 작업 후 변경 내용을 요약받기 좋았다.

Codex는 특히 내가 이미 방향을 정했고, 실제 코드를 바꿔야 하는 상황에서 좋았다. 생각을 오래 넓히는 것보다, 정해진 목표를 코드베이스 안에서 처리하는 데 강했다.

## Claude Code의 장점

내 기준에서 Claude Code의 장점은 다음과 같았다.

- 터미널 중심으로 긴 개발 세션을 이어가기 좋았다.
- 처음 보는 코드베이스를 탐색하고 구조를 설명하게 하기 좋았다.
- Plan Mode로 큰 변경 전에 설계를 점검하기 좋았다.
- `CLAUDE.md`, memory, custom command, hook 같은 개인화 흐름이 강했다.
- worktree 기반 병렬 작업이나 장기 작업 관리 흐름이 잘 맞았다.
- `@파일` 참조로 분석 범위를 명확히 주기 좋았다.

Claude Code는 특히 "먼저 이해하고, 계획하고, 여러 단계로 진행하는 작업"에 잘 맞았다. 큰 리팩터링이나 테스트 보강처럼 변경 전 탐색이 중요한 작업에서 장점이 있었다.

## 같은 작업을 두 도구에 다르게 요청할 수 있었다

예를 들어 `/ai` 섹션을 블로그처럼 확장하는 작업이 있다고 하면 Codex에는 이렇게 요청할 수 있었다.

```txt
content/ai의 MDX를 읽어서 /ai 목록과 /ai/[slug] 상세 페이지를 만들어줘.
기존 /blog 구조를 참고하고, npm run build로 확인해줘.
```

이 요청은 구현 중심이었다. Codex가 파일을 만들고, 라우트를 추가하고, 빌드 결과까지 확인하기 좋은 형태였다.

Claude Code에는 같은 일을 조금 다르게 요청할 수 있었다.

```txt
Plan Mode로 먼저 /ai 섹션 구조를 설계해줘.

확인할 것:
- 기존 /blog 구조를 재사용할 수 있는지
- content/ai frontmatter에 필요한 필드
- 검색 대상에 AI 글을 포함하는 방법
- 모바일에서 목록 카드가 깨질 수 있는 지점

계획이 괜찮으면 그 다음 구현으로 넘어가자.
```

이 요청은 계획 중심이었다. Claude Code가 먼저 구조를 분석하고, 위험한 지점을 정리한 뒤 구현으로 넘어가기 좋은 형태였다.

## 프론트엔드 작업별 선택 기준

내 기준으로는 이렇게 나눠 쓸 수 있었다.

```txt
작고 명확한 기능 구현:
Codex

여러 파일에 걸친 기존 패턴 반영:
Codex 또는 Claude Code

처음 보는 코드베이스 탐색:
Claude Code

큰 리팩터링 전 계획 수립:
Claude Code Plan Mode

Next.js 라우트, MDX, 스타일 수정:
Codex

테스트 보강과 실패 원인 추적:
Claude Code 또는 Codex

반복 명령, hook, custom command:
Claude Code

프로젝트별 고정 규칙:
Codex는 AGENTS.md, Claude Code는 CLAUDE.md
```

이 기준은 절대적인 정답은 아니었다. 하지만 작업을 "탐색", "계획", "구현", "검증", "기록"으로 나누면 어떤 도구를 써야 할지 더 선명해졌다.

## 둘 다 조심해야 할 점

두 도구 모두 실제 코드를 바꿀 수 있기 때문에 검증 습관이 중요했다.

- 변경된 diff를 직접 읽어야 했다.
- 요청하지 않은 파일이 바뀌었는지 확인해야 했다.
- 새 의존성이 들어갔다면 이유를 확인해야 했다.
- 빌드, 타입 체크, 테스트를 실행해야 했다.
- UI 변경은 모바일까지 확인해야 했다.
- 이해하지 못한 코드는 그대로 합치면 안 되었다.

AI가 코드를 잘 만든 것처럼 보여도, 프로젝트의 책임 구조와 사용자 경험까지 자동으로 보장해주지는 않았다. 결국 최종 판단은 개발자의 몫이었다.

## 결론

Codex와 Claude Code는 둘 다 개발 생산성을 높여주는 코딩 에이전트였다. 하지만 잘 쓰려면 같은 방식으로 다루면 안 되었다.

Codex는 목표가 분명한 구현, 수정, 빌드 확인 흐름에 잘 맞았다. Claude Code는 터미널 중심의 긴 세션, 코드베이스 탐색, 계획 수립, memory와 custom workflow를 활용하는 흐름에 잘 맞았다.

프론트엔드 개발에서는 두 도구 모두 강력했다. React와 React Native 작업은 파일, 상태, 라우트, 스타일, 테스트가 서로 연결되어 있기 때문이다. 중요한 것은 AI에게 맡길 일을 잘게 나누고, context를 정확히 주고, 결과를 직접 검증하는 태도였다.

AI를 잘 쓰는 개발자는 AI에게 모든 판단을 넘기는 사람이 아니었다. AI가 빠르게 만든 결과를 읽고, 의심하고, 실행해보고, 프로젝트에 맞는 선택으로 바꾸는 사람이었다.
